## Feature 001 — Структура проекта

**Задача:**
Обеспечить быстрый и предсказуемый способ добавления новых компонентов в проекте Feedium

**Проблема:**
- Отсутствие структуры проекта приводит к увеличению времени добавления новых компонентов (2–4 часа вместо ≤30 минут) и необходимости изменять несколько частей системы.

**Контекст:**
Начинается разработка MVP сервиса Feedium — персонального агрегатора контента из Telegram, RSS и веб-источников с последующим AI-ранжированием.

На старте проекта отсутствуют зафиксированные правила организации кода, при этом система предполагает развитие нескольких независимых частей (сбор данных, API, обработка, интерфейсы), которые будут развиваться параллельно.

Issue: https://github.com/4itosik/feedium/issues/1

**Для кого:**
Backend-разработчики проекта и будущие участники разработки

**Ожидаемый результат:**
разработчик может предсказуемо добавлять новые компоненты, не изменяя существующую логику, следуя документации проекта

**Ограничения:**
- Не реализовывать бизнес-логику (коллекторы, скоринг, ранжирование и т.д.)
- Не добавлять интеграции с внешними сервисами (Telegram, RSS, AI и др.)
- Не вводить новые доменные модели

**Критерий успеха:**
- Проект собирается командой `go build ./...` без ошибок
- приложение запускается командой (`go run ./cmd/feedium`) и завершается без ошибок на этапе старта
- В репозитории есть документация с описанием подхода к организации кода и правил взаимодействия частей системы
- Разработчик, не участвовавший в первоначальной реализации, может добавить новый компонент по документации за ≤30 минут
- Добавление нового компонента не требует изменения более чем 1–2 существующих файлов
- Изменения в одном компоненте не требуют модификации других компонентов (за исключением заранее определённых точек интеграции)

### Кратко о реализации

Заложена базовая структура проекта: `cmd/feedium` (entrypoint), `internal/app/`, `internal/bootstrap/`, `internal/platform/`, `internal/components/`. Настроен логгер (slog), написана документация по организации кода (ARCHITECTURE.md, PROJECT.md). Проект собирается и запускается без ошибок.

## Feature 002 — Health Check Endpoint

**Задача:**
Сделать доступной автоматическую проверку доступности сервиса внешними системами

**Проблема:**
Внешние системы (мониторинг, балансировщики) не могут выполнять HTTP-запрос к сервису и получать ответ о его состоянии, что делает невозможной настройку автоматических проверок и алертов

**Для кого:**
DevOps-инженеры проекта Feedium и инфраструктурные компоненты (системы мониторинга и балансировщики), выполняющие автоматические проверки доступности сервиса

**Контекст:**
Разрабатывается MVP сервиса Feedium.

На текущем этапе:
- отсутствуют внешние зависимости
- отсутствует база данных
- требуется базовый механизм проверки доступности сервиса для последующей интеграции с инфраструктурой

Issue: https://github.com/4itosik/feedium/issues/2

**Ограничения:**
- Проверка должна учитывать только доступность самого сервиса (без зависимостей)
- Не должна включать бизнес-логику
- Не должна требовать наличия базы данных или внешних сервисов

**Критерий успеха:**
- Внешняя система (например, мониторинг или балансировщик) может выполнить запрос к сервису и получить предсказуемый ответ, по которому можно определить доступность сервиса

### Кратко о реализации

Добавлен Connect-go, создан `GET /healthz` endpoint с ответом `{"status":"ok"}`. Реализован `bootstrap.Run(ctx, log)` с HTTP-сервером, graceful shutdown по SIGINT/SIGTERM. Написан unit-тест на хендлер. Обновлён `cmd/feedium/main.go` с signal handling.

## Feature 003 — Source Management API

**Задача:**
Сделать механизм управления списками источников для сбора данных

**Проблема:**
В системе отсутствует способ управления источниками данных, из-за чего любые изменения (добавление, редактирование, удаление) требуют изменения кода и деплоя.

**Для кого:**
Внутренние сервисы и разработчики через API (на этапе MVP — без аутентификации)

**Контекст:**
Feedium агрегирует данные из различных источников (Telegram, RSS и др.).
С ростом количества источников увеличивается необходимость частых изменений, что усиливает зависимость от разработки

- Количество источников будет измеряться от 10 до 1000
- Частота изменений низкая
- Типа источников заранее известны:
  - каналы в Telegram
  - группы в Telegram
  - RSS фиды с сайтов
  - Парсинг html сайтов
- обязательно валидация входных данных

**Ограничения:**
- Не требуется реализация integration и e2e тестов на данном этапе
- Без аутентификации и авторизации на данном этапе
- Управление касается только конфигурации источников
- Не включает выполнение сбора данных
- Не включает проверку доступности источников
- Не требуется пользовательский интерфейс (только API)

**Критерий успеха:**
Добавление, изменение и удаление источников не требует изменения кода и деплоя
Любой пользователь системы может через API:
- создавать источник данных
- редактировать источник данных
- удалять источник данных
- получать список всех источников

### Кратко о реализации

Полный CRUD + List для `Source` через Connect-go API. Proto-контракт (`api/source/v1/source.proto`), доменный слой (`internal/app/source/`), Postgres-репозиторий через GORM с JSONB для config, Connect-хендлер с маппингом ошибок. Строгая валидация по типу источника (telegram_channel, telegram_group, rss, web_scraping). Unit-тесты сервиса на mockgen, тесты хендлера и postgres-адаптера на go-mocket. Покрытие >80%.

## Feature 006 — Posts Management API

**Задача:**
Обеспечить возможность создавать, изменять, удалять и получать посты без изменения кода и деплоя

**Проблема:**
В системе отсутствует хранилище постов, из-за чего обработка данных невозможна (посты не сохраняются и не могут быть переданы в последующие этапы обработки)

**Для кого:**
Backend модули (другие пакеты) внутри проекта Feedium или разработчики, использующие API (на этапе MVP — без аутентификации)

**Контекст:**
Feedium агрегирует посты из различных источников (Telegram, RSS и др.),
поэтому необходимо иметь возможность управлять списком постов, которые были собраны.

- Количество постов будет измеряться от 100 до 10000 в день
- На этапе MVP обрабатываем только текстовые посты
- В системе создаются только посты, содержащие обязательные поля (текст, источник, дата публикации, meta информация (например автор))
- После создания пост должен быть доступен для чтения другими компонентами системы через API или внутренние интерфейсы
- К каждому посту обзяательно сохранять источник, из которого он был собран и метаданные (например, дату публикации, автора и т.д.)

**Ограничения:**
- Не требуется реализация integration и e2e тестов на данном этапе
- Без аутентификации и авторизации на данном этапе
- Не включает выполнение сбора данных
- Не включает проверку доступности источников
- Не требуется пользовательский интерфейс (только API)
- Должна поддерживаться фильтрация по дате создания при получении списка постов для ограничения объема данных

**Критерий успеха:**
Добавление, изменение и удаление постов выполняется через API без изменения кода и деплоя, время выполнения операций создания, изменения, удаления и получения списка постов ≤ 5 секунды при нагрузке до 10 000 постов в день
Любой клиент API (в рамках MVP без аутентификации) или через внутренние процессы:
- создавать новый пост
- редактировать пост
- удалять пост
- получать список всех постов

### Кратко о реализации

Полный CRUD + List для `Post` через Connect-go API. Миграция `002_create_posts.sql` с FK на `sources.id`, индексами. Доменный слой (`internal/app/post/`), Postgres-репозиторий через GORM с FK violation detection (код 23503), Connect-хендлер. Фильтрация по `published_at` в List. Unit-тесты сервиса, хендлера и postgres-адаптера с покрытием >80%. Подключение в bootstrap.

## Feature 007 — Дифференцированный процессинг материалов

**Задача:**
Обеспечить возможность автоматической обработки сохранённых материалов с учётом того, что для части из них результат доступен сразу, а для другой части — только раз в 12 часов или может быть инициирована пользователем через явное действие (ручной запуск).

**Проблема:**
Пользователи системы не получают результат обработки материалов в ожидаемое время или получают некорректный (неполный) результат.

**Причина:**
Момент, когда контекст материала является достаточным для корректной обработки, заранее не определён и зависит от типа материала и его источника.

**Следствие:**
- Часть материалов обрабатывается слишком рано → результат неполный
- Часть материалов не обрабатывается вовсе → пользователь не получает ценность

**Для кого:**

*Primary user:*
Пользователь, который сохраняет материалы и ожидает получить по ним результат обработки (например, summary) в предсказуемое время и с достаточной полнотой.

*Secondary:*
Разработчики и операторы системы, которым важно предсказуемое и расширяемое поведение обработки при добавлении новых источников и типов материалов.

**Контекст:**
Система сохраняет материалы из различных источников (например: Telegram, веб-сайты, Hacker News).

Основная ценность продукта — преобразование сохранённых материалов в структурированный результат (например, summary).

Отсутствие или некорректность обработки напрямую влияет на:
- снижение retention
- снижение perceived value продукта

Система находится на этапе MVP и будет расширяться за счёт новых источников и типов материалов.

**Ключевые термины:**

**Материал** — единица контента, сохранённая пользователем.
**Самодостаточный материал** — материал, содержащий полный смысловой контекст в момент сохранения.
**Накопительный материал** — материал, смысловой контекст которого формируется во времени за счёт поступления связанных данных.
**Достаточный контекст** — состояние материала, при котором дальнейшее поступление данных не приводит к существенному изменению результата обработки.
**Результат обработки** — структурированный вывод по материалу (например, summary). Формат определяется отдельно.

**Текущий процесс / Ожидаемая эволюция:**
Процессинга пока нет. Ожидаемая эволюция:

1. Сначала — два чётко выраженных типа: самодостаточные vs. накопительные.
2. Затем — появление новых источников (уже сейчас: Telegram, сайты, Hacker News).
3. Со временем — у одного источника может появиться несколько режимов (статья на HN — самодостаточна, тред с комментариями — накопителен).

**Риск наивного подхода:** единая логика запуска обработки для всех типов приведёт к тому, что накопительные материалы будут обрабатываться с неполным контекстом, а самодостаточные — с искусственной задержкой.

**Ожидаемый результат:**

1. Каждый материал обрабатывается в момент, когда его контекст является достаточным для получения корректного результата.
2. Время получения результата обработки предсказуемо и зависит от типа материала.
3. Пользователь может инициировать обработку материала вручную.
4. Добавление новых источников и режимов обработки не влияет на поведение уже существующих типов материалов.

**Критерии результата:**
- [ ] Для самодостаточного материала результат обработки появляется без ожидания дополнительных данных.
- [ ] Для накопительного материала обработка запускаается раз в 12 часов по умолчанию или может быть инициирована пользователем через явное действие (ручной запуск через API).
- [ ] Добавление нового источника или нового режима обработки не изменяет наблюдаемое поведение уже существующих сценариев.
- [ ] Тип материала определяется на основе типа источника

**Ограничения:**
- Типы источников из входа: Telegram-канал, сайт, Hacker News (статья + комментарии).
- Система должна поддерживать добавление новых типов источников без изменения поведения существующих типов
- Каждый материал имеет один активный режим обработки в момент сохранения

**Out of scope:**
- Формат и содержание результата обработки (summary и другие типы)
- Пользовательский интерфейс отображения результатов
- Хранение и версионирование результатов обработки
- Обработка ошибок, ретраи и деградационные сценарии

**Допущения:**
- Тип материала определяется комбинацией источника и контекста (режима), а не только источником
- Критерии достаточности контекста могут различаться для разных типов материалов

### Кратко о реализации

Реализован outbox-паттерн для асинхронной обработки: таблицы `summaries`, `summary_posts`, `outbox_events` с миграцией. Доменный слой с `ProcessingMode` (SelfContained / Cumulative). Worker с поллингом outbox-событий, создание/обновление саммари через stub processor. Определён режим обработки по типу источника. Connect API для управления процессингом. Unit-тесты с покрытием >80%.

## Feature 008 — AI-саммари для постов и переписок

**Задача:**
Сократить время, необходимое пользователю для ознакомления с содержанием постов и переписок, при сохранении высокой точности понимания ключевого смысла

**Проблема:**
Пользователь вынужден читать полный текст постов и переписок, что приводит к значительным временным затратам и снижению эффективности потребления контента.

Факты:
- Средняя длина поста составляет ~1000 слов
- Пользователь читает около 10 постов за сессию
- Отсутствует инструмент быстрого ознакомления с содержанием

Следствия:
- значительное время уходит на чтение полного текста
- пользователь ограничен в количестве обрабатываемого контента
- снижается ценность агрегатора при росте объёма данных

Критически важно:
Пользователь ориентирован на **сохранение точности понимания**, так как контент используется далее (например, для формирования дайджестов)

**Для кого:**
Активные пользователи Feedium, потребляющие агрегированный контент из нескольких источников (Telegram, RSS и др.) и работающие с ним для дальнейшего анализа или переработки

**Контекст:**
Feedium агрегирует контент из различных источников, включая посты и групповые переписки.

Система развивается в сторону:
- увеличения количества источников
- роста объёма входящего контента
- последующей обработки контента (например, формирование дайджестов)

На текущем этапе отсутствует механизм, позволяющий быстро и точно понять содержание материалов без чтения полного текста.

Без решения:
- пользовательский сценарий масштабирования потребления контента становится неэффективным
- дальнейшая обработка (включая создание дайджестов) усложняется из-за перегрузки пользователя

**Ограничения:**
- Не учитываются вопросы аутентификации и авторизации
- Не рассматриваются integration и e2e тесты
- Не накладываются ограничения на техническую реализацию
- Требования не фиксируют конкретный способ реализации

**Критерий успеха:**
- Пользователь может понять ключевой смысл поста или переписки без чтения полного текста
- Сохраняется высокая точность передачи смысла (информация достаточна для дальнейшего использования в дайджестах)
- Время ознакомления с единицей контента сокращается минимум в 2 раза
- Пользователь может последовательно обрабатывать ≥10 постов за сессию без потери качества понимания

### Кратко о реализации

Реализован OpenRouter Processor для генерации саммари через LLM (openai-go клиент). Шаблоны промптов встроены через `go:embed`. Retry-логика в Worker с экспоненциальным backoff (1/2/4 мин, макс. 3 попытки). Разделение ошибок на транзитные и перманентные. WorkerRunner — изолированный компонент поллинга с graceful shutdown. Read-методы в SummaryRepository с агрегацией (array_agg) против N+1. Connect API для чтения саммари (`GetSummaryByPost`, `ListSummariesBySource`). Unit-тесты с покрытием >80%.

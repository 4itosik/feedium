# FT-004: Posts Management

Issue: https://github.com/4itosik/feedium/issues/6

## Problem

В системе отсутствует механизм персистентного хранения постов. Коллекторы собирают посты из различных источников (Telegram, RSS и др.), но посты не сохраняются и недоступны последующим этапам конвейера обработки.

Последствия бездействия:
- Без персистентности постов блокируются последующие фичи MVP: FT-003 (коллекторы) не имеет места для записи результата, модули скоринга и ранжирования не имеют источника входных данных — фича блокирует ≥3 последующих фичи roadmap.
- Невозможно отлаживать конвейер обработки и анализировать качество собранных данных без способа их просмотра.

## Task

Обеспечить персистентное хранение постов и доступ к ним для других модулей системы:
1. Новые посты, полученные коллекторами, должны сохраняться и быть доступны после перезапуска сервиса.
2. Последующие модули обработки должны иметь возможность получать сохранённые посты с ограничением выборки по дате создания.
3. Разработчик при отладке конвейера должен иметь возможность удалить или поправить ошибочно собранный пост, не останавливая работу сервиса.

Выбор транспорта, протокола и формата интерфейса — задача Design-фазы.

## Context

- Разрабатывается MVP Feedium; FT-001 (каркас проекта) и FT-002 (health-check, PostgreSQL) завершены — есть инфраструктура для хранения данных.
- Коллекторы (FT-003 и далее) будут создавать посты после сборки данных из источников — они ближайший потребитель фичи.
- На этапе MVP требуется хранить только текстовые посты. Пост должен содержать достаточно атрибутов, чтобы потребитель мог идентифицировать источник, время публикации и автора. Конкретный набор полей определяется в Design на основе требований коллекторов FT-003.
- Прогнозируемая нагрузка MVP: до 10 000 постов в день (верхняя граница оценки PRD-001, точная цифра не зафиксирована — для Design считать верхнюю границу).
- Ограничение выборки по дате создания необходимо, чтобы потребители не запрашивали весь архив при каждом обращении.

## Users

- **Коллекторы постов** — сохраняют посты сразу после сбора из источников.
- **Модули последующей обработки** — читают сохранённые посты для дальнейшей обработки.
- **Разработчики Feedium** — изменяют и удаляют отдельные посты при отладке конвейера.

## Scope

### In Scope

- Персистентное хранение постов, переживающее перезапуск сервиса.
- Создание нового поста коллектором.
- Получение списка постов потребителем с ограничением выборки по дате создания.
- Изменение атрибутов ранее сохранённого поста (для исправления ошибочно собранных данных).
- Удаление поста по идентификатору.
- Автотесты операций согласно testing-policy проекта.

### Out Of Scope

- Аутентификация и авторизация.
- Пользовательский интерфейс (UI).
- Поиск и фильтрация по произвольным полям (кроме даты создания).
- Обработка и проверка доступности источников (входит в FT-003).
- Полнотекстовый поиск и индексирование.
- Сборка постов из источников (входит в коллекторы).

## Acceptance Criteria

- Созданный пост сохраняется и доступен для чтения после перезапуска сервиса.
- Пост содержит достаточно атрибутов, чтобы потребитель мог идентифицировать источник, время публикации и автора (конкретный состав — Design).
- Потребитель может получить список постов, ограниченный интервалом дат создания.
- Разработчик может изменить или удалить отдельный пост по идентификатору; результат виден при последующем чтении.
- Коллекторы и модули последующей обработки могут выполнять операции над постами из кода Feedium.
- Операции над постами (создание, чтение, изменение, удаление) покрыты автотестами согласно testing-policy проекта.

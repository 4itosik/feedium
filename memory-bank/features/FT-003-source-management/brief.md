# FT-003: Source Management

## Problem

В системе отсутствует способ управления конфигурацией источников сбора данных: любое добавление, изменение или удаление источника требует изменения кода и повторного деплоя сервиса.

Последствия бездействия:
- Каждое изменение состава источников блокируется полным циклом разработки (правка кода → review → сборка → деплой) вместо рантайм-операции.
- Коллекторы (FT-008) и остальные downstream-фичи не имеют источника правды о списке источников — их невозможно запустить без хардкода.
- Разработчики Feedium вынуждены тратить инженерное время на рутинные правки конфигурации, которые не требуют программирования по существу.

## Task

Дать разработчикам и внутренним сервисам Feedium возможность менять состав источников сбора данных в рантайме, без изменения кода и повторного деплоя сервиса.

Способ реализации (транспорт, формат запросов/ответов, набор операций, схема хранения, валидаторы) — предмет следующего этапа (spec), а не этого brief.

## Context

- Разрабатывается MVP Feedium; FT-001 (каркас проекта) и FT-002 (health-check + первичная настройка PostgreSQL) завершены — есть точка входа, DI, lifecycle и доступная БД для хранения конфигурации источников.
- Source — доменная сущность верхнего уровня в Feedium: от неё зависят коллекторы (FT-008), пайплайн обработки постов (FT-005, FT-006) и суммаризация (FT-007). FT-003 закрывает базовую CRUD-функциональность, без которой остальные фичи неработоспособны.
- Типы источников известны заранее и фиксированы на этапе MVP:
  - Telegram-канал
  - Telegram-группа
  - RSS-фид сайта
  - HTML-парсинг сайта
- Режим обработки (`self-contained` vs `cumulative`) определяется типом источника согласно `BR-01` в [PRD-001](../../prd/PRD-001-mvp.md) и задаётся системой, а не пользователем.
- Ожидаемый масштаб: от 10 до 1000 источников, частота изменений низкая (не чаще нескольких раз в день).
- Невалидные входные данные отклоняются до изменения состояния системы.

## Users

- **Разработчики Feedium** — носители боли: сегодня вынуждены править код и выкатывать деплой ради любого изменения состава источников. Основные пользователи управления источниками на этапе MVP.
- **Внутренние сервисы Feedium** (будущие коллекторы и планировщики) — потребители данных: читают актуальный список источников для запуска сбора.

На этапе MVP аутентификация и авторизация не требуются: управление доступно без идентификации вызывающего.

## Scope

### In Scope

- Создание нового источника с указанием типа (Telegram-канал, Telegram-группа, RSS, HTML-парсинг) и параметров, необходимых для его идентификации.
- Изменение параметров существующего источника.
- Удаление источника.
- Получение полного списка зарегистрированных источников.
- Валидация входных данных: корректность типа источника и обязательных полей для каждого типа.
- Персистентное хранение конфигурации источников (выбор конкретного хранилища — часть реализации).

### Out Of Scope

- Аутентификация и авторизация вызовов.
- Выполнение самого сбора данных из источников (задача FT-008 «Коллекторы»).
- Проверка фактической доступности источника (жив ли Telegram-канал, отдаёт ли RSS валидный XML).
- Пользовательский интерфейс для управления источниками.
- Integration- и e2e-тесты на этом этапе (unit-тесты по политике [testing-policy](../../engineering/testing-policy.md) остаются обязательными).
- Управление любыми сущностями, кроме конфигурации источников (посты, саммари, скоры).
- Добавление новых типов источников сверх четырёх перечисленных.

## Acceptance Criteria

- Вызывающий может зарегистрировать источник любого из четырёх поддерживаемых типов, и созданный источник возвращается в последующем запросе на получение списка.
- Вызывающий может изменить параметры существующего источника, и изменения отражаются в последующем запросе на получение списка.
- Вызывающий может удалить источник, и удалённый источник отсутствует в последующем запросе на получение списка.
- Вызывающий может получить полный список зарегистрированных источников.
- Запрос с некорректным типом источника или отсутствующими обязательными полями отклоняется с диагностической информацией, достаточной для исправления ошибки; состояние системы не изменяется.
- Добавление, изменение и удаление источника не требует изменения кода или повторного деплоя сервиса.
- Управление источниками не выполняет сбор данных и не проверяет доступность внешних источников.

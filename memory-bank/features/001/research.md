# Research: Feedium project setup skeleton

## 1. Цель исследования

Подготовить основу для `spec.md` по задаче создания **минимального setup-каркаса** проекта Feedium.

На этом этапе задача не в реализации бизнес-логики и не в создании рабочих компонентов, а в том, чтобы:

- зафиксировать верхнеуровневую структуру репозитория,
- задать понятные архитектурные границы,
- минимизировать связность между пакетами,
- обеспечить успешную сборку и запуск проекта,
- зафиксировать архитектурные правила в документации.

---

## 2. Уточнённые вводные

Приняты следующие решения:

1. **Интерфейсы объявляются в пакете-потребителе, а не в пакете-реализации**
   - цель: минимизировать зависимости между пакетами
   - избегать навязывания реализации

2. **Текущая задача — только setup проекта**
   - без бизнес-логики
   - без инфраструктуры
   - без интеграций

3. **Минимальное поведение при запуске**
   - `main.go` выводит:
     ```
     Feedium is starting
     ```

4. **Нужно создать**
   - каркас директорий
   - `ARCHITECTURE.md`
   - без реализации внутренних слоёв

---

## 3. Scope исследования

На этом этапе исследуем:

- минимальный корректный layout Go-проекта
- какие части архитектуры нужно зафиксировать сразу
- какие части отложить
- как минимизировать связность

Не исследуем:

- бизнес-логику
- инфраструктуру
- взаимодействие компонентов
- интеграции

---

## 4. Минимальный layout Go-проекта

Go не требует строгой структуры, но есть устойчивые практики:

- `cmd/` — entrypoints
- `internal/` — внутренний код
- `go.mod` — модуль

### Вывод

Минимально достаточно:

cmd/feedium/main.go
internal/
api/
docs/
migrations/

---

## 5. Роль директорий (на этапе setup)

На этом этапе директории — это **декларация архитектуры**, а не реализация.

### cmd/
- entrypoint приложения
- только запуск
- без логики

### internal/
- весь внутренний код
- не предназначен для внешнего использования

### internal/platform/
- будущая инфраструктура (config, logger, db)
- сейчас — только обозначение зоны

### internal/components/
- будущие runtime-компоненты
- сейчас — только структура

### internal/app/
- будущая бизнес-логика
- сейчас — только структура

### api/
- будущие контракты (proto/openapi)
- без реализации

### docs/
- документация
- включает `ARCHITECTURE.md`

### migrations/
- будущие миграции
- пока пусто

---

## 6. Главный принцип: минимизация зависимостей

### Ключевое правило

> Интерфейсы объявляются в пакете, который их использует

### Следствия

- реализация не диктует контракт
- пакеты не тянут лишние зависимости
- проще менять реализацию
- проще тестировать

### Запрещено

- создавать “общие интерфейсы” заранее
- объявлять интерфейсы “на будущее”
- делать interface-first без потребителя

---

## 7. Что должно быть в main.go

Минимальный допустимый код:

- лог старта
- отсутствие зависимостей от других пакетов
- отсутствие логики

### Требование

При запуске:

`Feedium is starting` выводится в log

### Запрещено

- бизнес-логика
- инициализация сервисов
- работа с БД
- сложный bootstrap

---

## 8. Bootstrap — нужен или нет

На этом этапе:

- можно создать папку `internal/bootstrap/`
- но **не реализовывать её**

### Вывод

Bootstrap сейчас — это **декларация**, а не код.

---

## 9. Что НЕ должно быть реализовано

Строго запрещено на этом этапе:

- бизнес-логика
- сервисы
- репозитории
- интеграции (Telegram, RSS, AI)
- БД
- HTTP API
- lifecycle management
- component runner

---

## 10. Критерий достаточного результата

Setup считается успешным, если:

- проект собирается: `go build ./…`
- проект запускается: `go run ./cmd/feedium`
- при запуске выводится: `Feedium is starting`
- структура директорий создана
- есть `ARCHITECTURE.md`
- нет фейковой реализации

---

## 11. Риски

### 11.1. Преждевременная детализация

Если создать много файлов:

- появится иллюзия готовой архитектуры
- усложнится понимание

### Решение

Создавать только:

- директории
- минимальный код
- документацию

---

### 11.2. Неправильное использование интерфейсов

Если объявлять интерфейсы в реализации:

- растёт связанность
- ухудшается гибкость

---

### 11.3. Смешение setup и реализации

Если добавить “немного логики”:

- нарушится цель этапа
- усложнится дальнейший spec

---

## 12. Что должно быть в ARCHITECTURE.md

Документ должен описывать:

### 12.1. Назначение директорий

- где что находится
- где будет находиться в будущем

### 12.2. Правило интерфейсов

- объявляются у потребителя

### 12.3. Ограничения текущего этапа

- только skeleton
- без реализации

### 12.4. Правило main.go

- только старт
- без логики

---

## 13. Вывод

На этапе setup:

- не нужна “идеальная архитектура”
- нужна **понятная структура + правила**

Ключевые решения:

- использовать `cmd/` и `internal/`
- зафиксировать архитектуру через папки
- описать правила в `ARCHITECTURE.md`
- не писать лишний код
- не реализовывать поведение
- минимизировать зависимости через правильное использование интерфейсов

---

## 14. Главный принцип этапа

> Мы создаём не систему, а каркас системы

И этот каркас должен:

- быть понятным
- быть минимальным
- не ограничивать будущие решения
- не создавать ложной сложности

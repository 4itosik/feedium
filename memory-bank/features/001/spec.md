# Feedium Project Setup Skeleton

## Цель

Создать каркас проекта, содержащий только перечисленные директории и файлы, без реализации бизнес-логики, обеспечивающий успешную сборку и запуск приложения.

Research: memory-bank/features/001/research.md

## Scope

### Входит:
- Создание структуры директорий проекта
- Создание `go.mod`
- Создание минимального `main.go`
- Создание пакета `internal/platform/logger`
- Инициализация логгера через `log/slog`
- Добавление `ARCHITECTURE.md` в корень проекта
- Фиксация архитектурных правил

### НЕ входит:
- Реализация бизнес-логики
- Создание компонентов, сервисов, репозиториев
- Интеграции (Telegram, RSS, AI и др.)
- Работа с БД и API
- Реализация bootstrap/lifecycle (кроме инициализации логгера)

## Требования

### 1. Структура проекта

Проект должен содержать строго следующие директории:

- cmd/
- cmd/feedium/
- internal/
- internal/app/
- internal/components/
- internal/platform/
- internal/platform/logger/
- internal/bootstrap/
- api/
- migrations/

Проект должен содержать строго следующие файлы:

- cmd/feedium/main.go
- internal/platform/logger/logger.go
- ARCHITECTURE.md
- go.mod

Создание дополнительных файлов и директорий запрещено.

---

### 2. main.go

Файл `cmd/feedium/main.go` должен:

- быть единственной точкой входа
- не содержать бизнес-логики
- импортировать только:
  - стандартную библиотеку
  - `internal/platform/logger`
- инициализировать логгер через пакет `logger`
- записывать в stdout лог-сообщение, содержащее подстроку `Feedium is starting`

---

### 3. Директории и файлы

- Все директории создаются без реализации
- В проекте допускаются только следующие файлы:
  - `cmd/feedium/main.go`
  - `internal/platform/logger/logger.go`
  - `ARCHITECTURE.md`
  - `go.mod`

---

### 4. ARCHITECTURE.md

Файл должен содержать:

- назначение каждой директории
- правила взаимодействия пакетов
- правило: интерфейсы объявляются в пакете-потребителе
- ограничение текущего этапа (Project Setup)
- пошаговую инструкцию добавления нового компонента

---

### 5. Архитектурные правила

- Интерфейсы объявляются только в пакете, который их использует
- Запрещено создавать интерфейсы без использования

---

### 6. Logger (platform layer)

Должен быть создан пакет:

internal/platform/logger/logger.go

Требования:

- пакет использует стандартную библиотеку `log/slog`
- пакет содержит ровно одну экспортируемую функцию Init()
- пакет не содержит бизнес-логики
- пакет не зависит от других внутренних пакетов
-  Init() возвращает не-nil *slog.Logger
- logger пишет в stdout
- формат: текстовый (slog.NewTextHandler)
- уровень логирования: INFO

Функция инициализации:

```go
func Init() *slog.Logger
```

## **Инварианты**

- Проект собирается через go build ./...
- Приложение запускается через go run ./cmd/feedium
- В проекте отсутствуют функции, кроме:
	- main() в main.go		
    - функции инициализации логгера в logger.go
- В проекте отсутствуют импорты пакетов из internal/*, за исключением:
    - main.go → internal/platform/logger
- В проекте отсутствует бизнес-логика. Бизнес-логикой считается любая логика, связанная с обработкой доменных сущностей, интеграциями, данными, конфигурацией приложения, lifecycle или bootstrap, кроме:
	  - вызова logger.Init()
	  - вызова логирования сообщения `Feedium is starting`
- Пакет logger не зависит от других внутренних пакетов

---

## **Acceptance Criteria**

- Репозиторий содержит только указанные директории и файлы
- В проекте есть go.mod
- Проект успешно собирается (go build ./...) завершился с exit code 0
- Проект запускается (go run ./cmd/feedium)  завершился с exit code 0
- В проекте существует internal/platform/logger/logger.go
- Пакет logger использует log/slog
- main.go импортирует internal/platform/logger
- main.go вызывает logger.Init() ровно 1 раз
- При запуске приложение записывает в stdout лог-сообщение, содержащее подстроку `Feedium is starting`
- В корне проекта есть ARCHITECTURE.md
- Для каждой директории из раздела "Структура проекта" в ARCHITECTURE.md есть отдельный абзац с описанием назначения, минимум 1 предложение на директорию
- В ARCHITECTURE.md зафиксированы архитектурные правила
- В ARCHITECTURE.md есть инструкция добавления компонента
- В проекте отсутствуют функции, кроме допустимых
- В проекте отсутствуют запрещённые импорты

---

## **Ограничения**

На этапе Project Setup запрещено:
- добавлять бизнес-логику
- создавать компоненты с поведением
- создавать зависимости между пакетами internal/*, кроме:
    - main.go → internal/platform/logger
- добавлять зависимости, отсутствующие в стандартной библиотеке Go
- создавать дополнительные реализации функций или пакетов, не требуемые данной спецификацией
- добавлять конфигурацию, БД, API
- реализовывать bootstrap/lifecycle

---

## **Definition of Done**  

Каркас считается завершённым, если:
- проект собирается и запускается без ошибок
- структура проекта соответствует требованиям
- отсутствуют дополнительные файлы и код
- ARCHITECTURE.md содержит описание директорий и правил
- все Acceptance Criteria выполнены
---
## **Error Scenarios**

- Если отсутствует go.mod → задача считается невыполненной    
- Если go build ./... возвращает ошибку → задача считается невыполненной
- Если отсутствует хотя бы одна директория или файл → задача считается невыполненной
- Если обнаружены лишние файлы → задача считается невыполненной
- Если есть функции вне разрешённого списка → задача считается невыполненной
- Если есть импорт из internal/*, кроме internal/platform/logger → задача считается невыполненной
- Если пакет logger зависит от других внутренних пакетов → задача считается невыполненной
- Если logger.Init() возвращает nil → задача считается невыполненной
- Если файл ARCHITECTURE.md существует, но:
	  - имеет размер 0 байт
	  - или не содержит текста
	  - или содержит менее 1 непустого предложения (строка длиной ≥ 10 символов без учета пробелов)
	→ задача считается невыполненной
- Если logger.Init() не использует slog.NewTextHandler → задача считается невыполненной
- Если logger.Init() не пишет в stdout → задача считается невыполненной
- Если logger.Init() создает логгер с уровнем, отличным от INFO → задача считается невыполненной
- Если logger.Info(...) вызывается до logger.Init() → задача считается невыполненной
- Если результат logger.Init() не используется для логирования → задача считается невыполненной
- Если main.go не вызывает logger.Info(...) с сообщением `Feedium is starting` → задача считается невыполненной
- Если main.go не вызывает logger.Init() → задача считается невыполненной

## **Главная идея**

Это не реализация системы, а минимальный каркас, фиксирующий архитектуру и позволяющий безопасно начинать разработку.

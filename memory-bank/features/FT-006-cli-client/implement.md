# Implementation Plan

## Steps

### Шаг 1. Регистрация зависимостей

- **Цель.** Зафиксировать ровно одну новую top-level зависимость `github.com/spf13/cobra` и добавить `sigs.k8s.io/yaml`, если её ещё нет в `go.mod` (NFR-01).
- **Действия.**
  1. Проверить `go.mod`/`go.sum` на наличие `github.com/spf13/cobra` и `sigs.k8s.io/yaml`.
  2. Добавить недостающие модули через `go get` (отдельно cobra, отдельно yaml при необходимости).
  3. Запустить `go mod tidy`.
- **Зависимости.** Нет.
- **Результат.** Обновлённые `go.mod` и `go.sum` содержат cobra и `sigs.k8s.io/yaml`; других новых top-level модулей не появилось.
- **Как проверить.**
  - `go build ./...` проходит.
  - `grep "spf13/cobra" go.mod` и `grep "sigs.k8s.io/yaml" go.mod` возвращают по одной строке.
  - Diff `go.mod` не содержит иных top-level добавлений.

### Шаг 2. Makefile target `feediumctl`

- **Цель.** Обеспечить сборку `bin/feediumctl` единой командой `make feediumctl` без изменения существующего target `build` (NFR-02).
- **Действия.**
  1. Добавить `feediumctl` в список `.PHONY` в `Makefile`.
  2. Добавить правило `feediumctl`, собирающее бинарь `bin/feediumctl` из `./cmd/feediumctl`.
  3. Убедиться, что существующий target `build` не затронут.
- **Зависимости.** Шаг 1 (модули зарегистрированы), Шаг 3 (каркас `./cmd/feediumctl` должен существовать, иначе `make feediumctl` упадёт на отсутствии пакета).
- **Результат.** В `Makefile` появился target `feediumctl`; серверный target `build` сохраняет прежнее поведение.
- **Как проверить.**
  - `make -n feediumctl` печатает команду `go build -o bin/feediumctl ./cmd/feediumctl`.
  - `make -n build` по-прежнему собирает `bin/feedium`.
  - `make feediumctl` (после зависимостей) создаёт исполняемый `bin/feediumctl`.

### Шаг 3. Каркас команды `feediumctl` и корневой cobra-command

- **Цель.** Создать `cmd/feediumctl/` с точкой входа и корневой командой cobra, без какой-либо бизнес-логики (CON-01, INV-01).
- **Действия.**
  1. Завести пакет `main` в `cmd/feediumctl/` с функцией `main`, которая запускает корневую cobra-команду и по любой ошибке печатает её в stderr + `os.Exit(1)`.
  2. Создать корневую команду `feediumctl` с кратким описанием утилиты.
  3. Зарегистрировать глобальные persistent-флаги: `--endpoint` (string), `--output`/`-o` (string), `--timeout` (Go `time.Duration`), `--page-size` (int), `--config` (string). Значения по умолчанию на уровне cobra НЕ задавать — дефолты применяются в резолвере (Шаг 5).
  4. Убедиться, что `cmd/feediumctl/...` не импортирует `internal/biz`, `internal/data`, `internal/service`.
- **Зависимости.** Шаг 1, Шаг 2.
- **Результат.** Рабочий каркас CLI, отображающий usage и принимающий глобальные флаги.
- **Как проверить.**
  - `make feediumctl && ./bin/feediumctl --help` → exit `0`, stdout содержит usage, stderr пуст, RPC не вызывается (INV-03).
  - `grep -R "internal/\(biz\|data\|service\)" cmd/feediumctl/` пуст (AC-05).

### Шаг 4. Загрузка YAML-конфига

- **Цель.** Реализовать чтение конфиг-файла по правилам FR-06/EC-D/EC-E/EC-G с закрытым набором ключей `endpoint`, `output`, `timeout`, `page_size`.
- **Действия.**
  1. Завести внутренний подпакет в `cmd/feediumctl/` (например, `cmd/feediumctl/internal/config`), отвечающий за резолвинг пути и парсинг YAML.
  2. Реализовать определение пути: `--config` > env `FEEDIUMCTL_CONFIG` > `~/.feediumctl.yaml`. Флаг и env помечаются как «явно указанный путь» — отличие от дефолта хранится вместе с путём.
  3. Парсить YAML строгим режимом: неизвестные ключи → ошибка. Допускаются только четыре ключа. Тип `timeout` — строка, парсится через `time.ParseDuration` на этом же шаге; `page_size` — целое число; `output` — строка; `endpoint` — строка. Любая ошибка парсинга конвертируется в ошибку категории `config`.
  4. Реализовать поведение «нет файла» по источнику пути:
     - Дефолтный путь и `$HOME` не резолвится / файла нет → silent fallback (пустой конфиг, без ошибки).
     - Явный путь (флаг/env) и файла нет → ошибка `config: <path>: not found`.
  5. Формат всех ошибок этого шага — одна строка `config: <path>: <reason>` (NFR-03).
- **Зависимости.** Шаг 1, Шаг 3.
- **Результат.** Функция загрузки конфига возвращает нормализованные значения (или их отсутствие) и ошибку с префиксом `config`.
- **Как проверить.**
  - Unit-тесты на: отсутствующий дефолтный файл, отсутствующий явный файл, неразбираемый YAML, неизвестный ключ, валидный полный файл, валидный файл с подмножеством ключей, `$HOME` не определён.
  - В каждом error-кейсе сообщение совпадает с шаблоном `config: <path>: <reason>` и exit-код в CLI равен `1`.

### Шаг 5. Резолвер настроек по приоритету `flag > env > config > default`

- **Цель.** Определить эффективные значения `endpoint`, `output`, `timeout`, `page_size` по единому правилу приоритетов (INV-05, FR-02..05).
- **Действия.**
  1. Для каждого параметра реализовать резолв: если cobra сообщает, что флаг установлен пользователем — берём его; иначе env; иначе конфиг; иначе дефолт.
  2. Дефолты: `endpoint=localhost:9000`, `output=table`, `timeout=1m`, `page_size=50`.
  3. Env-переменные: `FEEDIUMCTL_ENDPOINT`, `FEEDIUMCTL_OUTPUT`, `FEEDIUMCTL_TIMEOUT`, `FEEDIUMCTL_PAGE_SIZE`.
  4. Ошибки парсинга env/flag-значений (кроме `--output`):
     - невалидный `timeout` → `flag: invalid timeout "<raw>": <reason>`;
     - невалидный `page_size` → `flag: invalid page size "<raw>": <reason>`;
     - невалидное значение флага, относящегося к иным параметрам → `flag: <reason>`.
  5. Валидация значения `output` вынесена в Шаг 6.
- **Зависимости.** Шаг 3 (флаги), Шаг 4 (конфиг).
- **Результат.** Структура с эффективными значениями четырёх параметров и единый источник ошибок резолва с префиксами `config`/`flag`.
- **Как проверить.**
  - Unit-таблица `flag > env > config > default` по всем четырём параметрам (AC-03).
  - Отдельные тесты на невалидные `--timeout=abc`, `--page-size=-1` через env и флаг: сообщение соответствует префиксу `flag:` (NFR-03).

### Шаг 6. Валидация формата вывода

- **Цель.** Гарантировать, что `output ∈ {table, json, yaml}` с фиксированным сообщением об ошибке (FR-04, EC-B).
- **Действия.**
  1. После резолва (Шаг 5) проверить значение `output`.
  2. При неизвестном значении вернуть ошибку текста `output: invalid value "<raw>" (allowed: table,json,yaml)` и завершить работу с exit `1`, не открывая gRPC-соединение.
  3. Префикс `output` используется ИСКЛЮЧИТЕЛЬНО для нарушений FR-04.
- **Зависимости.** Шаг 5.
- **Результат.** Команды не выполняют RPC при неверном output; stderr-сообщение детерминировано.
- **Как проверить.**
  - Unit-тест: `--output=xml` → точный текст сообщения и exit `1`.
  - Тесты источников: то же значение через env и конфиг ведёт к той же ошибке.

### Шаг 7. Единый репортёр ошибок и exit-логика

- **Цель.** Обеспечить шаблоны NFR-03, INV-02 и FR-11 (stdout пуст при ошибке, exit `1`, нет stacktrace).
- **Действия.**
  1. Реализовать функцию вывода ошибки с закрытым списком префиксов `config`, `flag`, `output`, `endpoint`. Любая локальная ошибка формируется как `<prefix>: <reason>` в одну строку.
  2. Реализовать форматтер RPC-ошибок: из `status.FromError` взять `codes.Code` и `Message()`, вывести `code=<CodeName> message=<msg>`. Имя кода — каноническое (`codes.Code.String()`).
  3. Гарантировать: при ошибке stdout не пишется ничего; stderr получает ровно одну строку (без trailing stacktrace); exit `1` для всех категорий.
  4. Успех: stdout получает отформатированный ответ, stderr пуст, exit `0`.
- **Зависимости.** Шаги 4, 5, 6 (поставщики локальных ошибок). Шаг 8 потребляет форматтер этого шага, а не наоборот — обратной зависимости нет.
- **Результат.** Единая точка для всех ошибок CLI с предсказуемыми сообщениями.
- **Как проверить.**
  - Unit-тесты на каждое имя префикса из закрытого списка (AC-06).
  - Unit-тест на форматирование `codes.Unavailable`, `codes.DeadlineExceeded`, `codes.InvalidArgument`, `codes.Internal`.
  - Smoke: любая симулированная ошибка → stdout пуст, stderr ровно одна строка, exit `1` (INV-02).

### Шаг 8. gRPC-транспорт

- **Цель.** Открывать соединение с сервером через plaintext gRPC и корректно классифицировать ошибки стадии (FR-07, EC-A, EC-C).
- **Действия.**
  1. Реализовать фабрику gRPC-клиентов, принимающую эффективный endpoint и возвращающую соединение с `insecure.NewCredentials()` и без metadata-аутентификации.
  2. Перед dial выполнить синтаксическую проверку endpoint (например, поиск порта/отсутствие недопустимых символов). При провале — ошибка с префиксом `endpoint: <reason>` и exit `1`; RPC не выполняется.
  3. Ошибки фазы dial/RPC транслируются через форматтер из Шага 7 по шаблону `code=<CodeName> message=<msg>`. Для `codes.Unavailable` допустимо наличие подстроки `connection refused` в сообщении (EC-A).
  4. Гарантировать, что клиент закрывается (`Close`) в defer независимо от исхода.
- **Зависимости.** Шаги 5, 7.
- **Результат.** Единая точка создания gRPC-соединения для всех будущих команд; ошибки имеют корректный префикс в зависимости от стадии.
- **Как проверить.**
  - Ручной запуск при выключенном сервере → stderr соответствует `^code=Unavailable message=.*` или содержит `connection refused` (EC-A).
  - Запуск с `--endpoint=::::` → stderr либо `endpoint: <reason>`, либо `code=Unavailable …` (EC-C) в зависимости от стадии провала; exit `1`.

### Шаг 9. Рендерер ответов: JSON и YAML

- **Цель.** Реализовать общий детерминированный рендер proto-сообщения в `json` и `yaml` (FR-10, NFR-06, INV-06).
- **Действия.**
  1. Создать утилиту рендера в подпакете `cmd/feediumctl/...`, принимающую `proto.Message` и желаемый формат.
  2. Для JSON использовать `protojson.MarshalOptions{Multiline: true, Indent: "  ", UseProtoNames: true, EmitUnpopulated: false}`.
  3. Для YAML конвертировать результат protojson в YAML через `sigs.k8s.io/yaml.JSONToYAML`; ключи YAML должны получаться лексикографически отсортированными.
  4. Вывод завершать ровно одним `\n`; никаких дополнительных пустых строк.
  5. Рендерер пишет в переданный `io.Writer` (stdout из CLI).
- **Зависимости.** Шаг 1.
- **Результат.** Общая функция рендера JSON/YAML для любых proto-ответов; байт-в-байт стабильность при одинаковом входе.
- **Как проверить.**
  - Unit-тест: одинаковый вход → байт-в-байт одинаковый выход (INV-06).
  - Unit-тест: отсутствие `EmitUnpopulated` полей в JSON; snake_case-имена (`UseProtoNames`).
  - Unit-тест: ровно один trailing `\n`.

### Шаг 10. Рендерер ответа `health` в формате `table`

- **Цель.** Реализовать вывод `V1CheckResponse` в формате `table` по FR-09.
- **Действия.**
  1. Расширить рендерер (Шаг 9) ветвью `table`: для `V1CheckResponse` печатать две колонки `FIELD | VALUE` и одну строку `status | <value>`.
  2. Разделитель и порядок колонок — фиксированы (NFR-06); line ending — `\n`; trailing `\n` ровно один.
- **Зависимости.** Шаг 9.
- **Результат.** Команда `health` имеет детерминированный табличный вывод.
- **Как проверить.**
  - Unit-тест на байт-в-байт стабильность вывода `table` для фиксированного `V1CheckResponse`.
  - Unit-тест: два заголовка `FIELD` и `VALUE`, одна data-строка.

### Шаг 11. Команда `feediumctl health`

- **Цель.** Собрать сквозной сценарий FR-01: резолв настроек → dial → RPC → рендер.
- **Действия.**
  1. Зарегистрировать sub-command `health` у корневой команды.
  2. В обработчике: запустить резолвер (Шаг 5), валидацию output (Шаг 6), получить gRPC-соединение (Шаг 8), создать `context.WithTimeout` от эффективного `--timeout` (FR-05, INV-04), вызвать `HealthService.V1Check` с пустым `V1CheckRequest`.
  3. Ответ передать в рендерер (Шаги 9, 10) согласно эффективному `output`.
  4. Любая ошибка → Шаг 7; stdout пуст.
  5. Убедиться, что `feediumctl --help` и `feediumctl health --help` не открывают соединение и не вызывают RPC (INV-03).
- **Зависимости.** Шаги 5, 6, 7, 8, 9, 10.
- **Результат.** Рабочая команда `feediumctl health`.
- **Как проверить.**
  - AC-01: запуск против живого сервера с дефолтами — stdout с табличным ответом, exit `0`.
  - AC-02: запуск при выключенном сервере — exit `1`, stderr соответствует EC-A, stdout пуст.
  - AC-07: `feediumctl health --timeout=1ms` против заведомо медленного сервера — stderr matches `^code=DeadlineExceeded message=.*`, exit `1`.

### Шаг 12. Консолидация тестов и финальный прогон

- **Цель.** Убедиться, что unit-тесты, написанные в рамках Шагов 4–11, в сумме покрывают AC-03, AC-06 и инварианты INV-02/05/06; добавить тесты, не закрытые ни одним из предыдущих шагов.
- **Соглашение.** Тесты пишутся вместе со своими шагами (блок «Как проверить» каждого из Шагов 4–10 — это точка создания соответствующих тестов). Этот шаг их не дублирует, а сводит и дополняет.
- **Действия.**
  1. Сверить фактический набор тестов с карточкой покрытия:
     - AC-03 — таблица приоритетов для 4 параметров (пишется в Шаге 5);
     - AC-06 — парсинг конфига (Шаг 4), рендер трёх форматов (Шаги 9, 10), шаблоны ошибок для префиксов `config`/`flag`/`output`/`endpoint` (Шаги 4, 5, 6, 7, 8) и RPC-шаблон `code=… message=…` (Шаг 7);
     - INV-02 — smoke в Шаге 7; INV-05 — в Шаге 5; INV-06 — в Шагах 9, 10.
  2. Добавить недостающее (вероятно отсутствует ни в одном из предыдущих шагов):
     - тест формирования `V1CheckRequest` — проверка, что запрос пустой (FR-01);
     - интеграционный тест обработчика `health` против mock gRPC-клиента `HealthService` (без сетевых вызовов): happy-path и возврат `status.Error(codes.Unavailable, ...)` маршрутизируются через форматтер из Шага 7.
  3. Если какой-то пункт AC-03/AC-06 не покрыт ни одним существующим тестом — добавить тест в тот шаг, к которому он тематически относится, и обновить ссылку в «Как проверить» соответствующего шага.
- **Зависимости.** Шаги 4–11.
- **Результат.** Зелёный `go test ./cmd/feediumctl/... -race` с явным соответствием каждого AC/INV конкретному тесту.
- **Как проверить.**
  - `go test ./cmd/feediumctl/... -race` завершается успешно.
  - `go vet ./...` и существующие lint-проверки проходят (AC-05).
  - `grep -R "internal/\(biz\|data\|service\)" cmd/feediumctl/` пуст (INV-01, AC-05).

### Шаг 13. Финальная сборка и сквозная проверка

- **Цель.** Убедиться, что `make feediumctl` даёт работоспособный бинарь и acceptance-сценарии проходят end-to-end.
- **Действия.**
  1. Выполнить `make feediumctl` и проверить `bin/feediumctl`.
  2. Запустить сервер локально, выполнить сценарии AC-01, AC-04 и AC-07; зафиксировать stdout/stderr/exit.
  3. Остановить сервер и проверить AC-02.
- **Зависимости.** Все предыдущие шаги.
- **Результат.** Подтверждённые AC-01, AC-02, AC-04, AC-07 вручную; AC-03, AC-05, AC-06 — автоматически (Шаг 12).
- **Как проверить.**
  - `./bin/feediumctl --help` → exit `0`, usage в stdout (AC-04).
  - `./bin/feediumctl health` при живом сервере → exit `0`, табличный вывод (AC-01).
  - `./bin/feediumctl health` при выключенном сервере → exit `1`, stderr по EC-A, stdout пуст (AC-02).
  - `./bin/feediumctl health --timeout=1ms` при медленном/недоступном сервере → поведение EC-H (AC-07).

## Edge Cases

- **EC-A Сервер недоступен.** `feediumctl health` без запущенного сервера: stderr matches `^code=Unavailable message=.*` либо содержит `connection refused`; stdout пуст; exit `1`. Покрывается Шагами 7, 8, 11.
- **EC-B Неизвестный `--output`.** `--output=xml` (или любое значение вне `table|json|yaml`) из любого источника: stderr ровно `output: invalid value "xml" (allowed: table,json,yaml)`; RPC не выполняется; exit `1`. Покрывается Шагом 6.
- **EC-C Невалидный endpoint.** `--endpoint=::::`: либо stderr `endpoint: <причина>` (этап парсинга), либо `code=Unavailable …` (этап dial); exit `1`. Покрывается Шагом 8.
- **EC-D Дефолтный конфиг отсутствует.** Флаг/env не заданы, `~/.feediumctl.yaml` отсутствует или `$HOME` не определён: CLI работает на env/дефолтах, без предупреждений; exit определяется RPC. Покрывается Шагом 4.
- **EC-E Конфиг невалидный.** Неразбираемый YAML или неизвестный ключ (любой источник пути): stderr `config: <path>: <reason>`; exit `1`. Покрывается Шагом 4.
- **EC-F Несколько источников одного значения.** Флаг и env заданы одновременно — побеждает флаг; env и config — побеждает env; и т.д. Покрывается Шагом 5 (INV-05).
- **EC-G Явный конфиг отсутствует.** `--config=/missing.yaml` или `FEEDIUMCTL_CONFIG=/missing.yaml`: stderr `config: /missing.yaml: not found`; RPC не выполняется; exit `1`. Покрывается Шагом 4.
- **EC-H Таймаут истёк.** `--timeout=1ms` против медленного сервера: stderr matches `^code=DeadlineExceeded message=.*`; exit `1`; stdout пуст. Покрывается Шагами 7, 11.
- **Невалидный `--timeout` / `--page-size`.** Любой источник, кроме `output`: stderr `flag: <reason>`; exit `1`. Покрывается Шагом 5.
- **`--help` на любой команде.** stdout с usage; RPC не выполняется; exit `0` (INV-03). Покрывается Шагами 3, 11.
- **Успешный ответ.** stdout содержит рендер в выбранном формате с ровно одним trailing `\n`; stderr пуст; exit `0` (INV-02, FR-11, NFR-06).

## Open Questions

- **OQ-01 (из spec).** Стратегия версионирования CLI-команд при появлении `V2*`-методов в proto (`--api-version`, отдельные подкоманды, always-latest и break-change) — в MVP не решается, перенесено из spec.md без изменений.
- **OQ-02 Граница парсинга endpoint.** Spec (EC-C) допускает оба исхода — `endpoint: <причина>` на парсинге или `code=Unavailable …` на dial. Необходимо зафиксировать перечень синтаксических проверок, выполняемых в Шаге 8 перед `grpc.Dial`, чтобы поведение `--endpoint=::::` было воспроизводимым между запусками. В MVP допускается опираться на проверки, встроенные в gRPC dial, и оставить префикс `endpoint:` только для явно некорректных случаев (например, пустая строка).
- **OQ-03 Строгость YAML-парсера для `timeout`/`page_size`.** Spec не уточняет, допустимы ли нестандартные представления (`"30s"` vs `30s`, `"50"` vs `50`). Текущий план опирается на строгий парсинг ключей и стандартные типы Go; любые расширения форматов — правка spec.
- **OQ-04 Имя `implement.md` vs `implementation-plan.md`.** Каталог `memory-bank/features/FT-006-cli-client/` по feature-flow ожидает `implementation-plan.md`, но запрос пользователя — `implement.md`. Файл создан под именем `implement.md`; при необходимости синхронизировать с feature-flow — отдельный шаг, вне scope текущего плана.

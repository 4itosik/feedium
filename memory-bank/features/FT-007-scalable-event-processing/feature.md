# FT-007: Scalable Event Processing

## What

### Problem

FT-005 доставил суммаризацию, но обработчик событий построен в предположении «один процесс, одна горутина». Из-за этого сейчас:

1. **Пост-события обрабатываются строго по очереди.** Длинная обработка одного события (медленный LLM-ответ, long cumulative-суммаризация) блокирует все остальные пост-события, которые могли бы идти параллельно.
2. **Нельзя запустить более одного воркера — ни внутри процесса, ни между процессами.** Выдача событий происходит без защитного захвата на уровне БД, поэтому два наблюдателя подхватят одно и то же событие, выполнят двойную LLM-оплату и оставят систему в рассогласованном состоянии.
3. **Cumulative-суммаризация зависит от внутреннего тикера.** Сейчас один процесс периодически обходит все cumulative-источники списком; второй процесс этот обход дублирует. Нет per-source понятия «когда источник в следующий раз должен быть просуммирован», поэтому невозможно аккуратно регулировать нагрузку и невозможно масштабировать планировщик.

Последствия бездействия: FT-005 не выдерживает целевой нагрузки PRD-001 (до 10 000 постов в день — RISK-02) и блокирует rollout на несколько инстансов. Окно обработки растёт линейно с нагрузкой; деплой с zero-downtime (rolling update) невозможен, потому что одновременный запуск нового и старого процессов приводит к дублям.

### Outcome

| `MET-03` | Автоматическое восстановление «зависших» событий | Stuck в `processing` навсегда | Событие, осиротевшее при падении воркера, автоматически возобновляется | Интеграционный тест: симулируем crash после захвата, событие подхватывается другим воркером в ограниченное время |

### Scope

- `REQ-01` События обрабатываются по одному, с атомарным захватом воркером — исключается двойная обработка одного события двумя параллельными наблюдателями (внутри процесса или между процессами).
- `REQ-02` Число параллельных воркеров внутри одного процесса задаётся конфигом. Каждый воркер работает независимо, без центрального диспетчера.
- `REQ-03` Запуск нескольких процессов над одной БД корректен без какой-либо внешней координации: ни одно событие не обрабатывается дважды.
- `REQ-04` Воркер, который начал обработку события, удерживает его на ограниченное время (lease) и периодически продлевает lease. Если воркер упал, lease истекает и другой воркер обязан подхватить событие.
- `REQ-05` Transient-ошибки (сеть, таймаут LLM, временная недоступность внешнего сервиса) не превращают событие в окончательно проваленное: событие возвращается в очередь с экспоненциальной задержкой до ограниченного числа попыток.
- `REQ-06` Cumulative-источники планируются индивидуально: у каждого источника есть момент «когда в следующий раз пора суммаризировать». Планировщик не выполняет массовый обход списка и не дублирует работу между инстансами.
- `REQ-07` В системе существует механизм санации «зависших» событий: записи, у которых lease истёк и число попыток исчерпано, переводятся в терминальный статус без вмешательства человека.
- `REQ-08` Graceful shutdown: при остановке процесса уже начатое событие доводится до терминального состояния в ограниченное время; новые события не берутся.
- `REQ-09` Прежние механизмы (пакетный обход pending-событий и внутренний тикер cumulative-обхода) удалены после переноса логики, без «двойных» путей и dead-code.

### Non-Scope

- `NS-01` Вынос очереди событий во внешний брокер (Redis, NATS, RabbitMQ). Инфраструктура MVP ограничена PostgreSQL.
- `NS-02` Leader election между процессами и distributed coordination через consensus-сервис. Не требуется.
- `NS-03` Per-source настраиваемая частота суммаризации. Все cumulative-источники планируются с единым интервалом из конфига.
- `NS-04` Изменение retry-логики внутри одного события (повторы самого LLM-вызова). Они остаются как в FT-005; FT-007 добавляет повторы только на уровне события.
- `NS-05` Приоритизация событий (пост-события важнее source-событий или наоборот). Все захватываются в порядке готовности.
- `NS-06` Метрики, tracing, алертинг для новых воркеров — отдельная feature (Observability).

### Constraints / Assumptions

- `ASM-01` PostgreSQL поддерживает row-level примитив, гарантирующий, что из набора строк только один наблюдатель получает строку и блокирует её для остальных (без блокировки таблицы в целом). Механизм существует в Postgres с версии 9.5; у нас в продакшене 18.3.
- `ASM-02` Единственное хранилище состояния событий и источников — та же БД, что используется FT-005. Нет внешних очередей, кешей, координаторов.
- `ASM-03` По умолчанию число параллельных воркеров равно 2 — минимум, при котором конкурентная семантика (захват, lease, multi-worker race) реально упражняется на dev/test-окружении и не маскируется single-goroutine-путём. Рост конкурентности сверх дефолта — задача деплоя, не feature.
- `CON-01` Гарантия «не более одного активного события на post / source» (уникальные индексы FT-005) сохраняется как инвариант. Любое изменение обработчика не должно ослабить её.
- `CON-02` Идемпотентность FT-005 — повторная обработка одного события после crash-recovery не создаёт дубликат Summary для того же поста.
- `CON-03` Graceful shutdown обязан завершить уже сохранённую Summary переводом события в терминальный статус. Summary, записанная без перевода события, заблокирует повторный enqueue через уникальный индекс.

## How

### Solution

Заменить batch-обработчик и in-process тикер на pull-модель поверх той же БД: воркеры сами атомарно захватывают одну единицу работы (событие или cumulative-источник), удерживают её через lease и отпускают по окончании. Санация зависших событий — отдельная фоновая задача. Главный trade-off: в схему `summary_events` и `sources` добавляется служебное состояние (lease и счётчик попыток, `next_summary_at`), но взамен система становится масштабируемой без внешних зависимостей.

### Change Surface

| Surface | Why it changes |
| --- | --- |
| Schema `summary_events` | Требуется поле lease и счётчик попыток с отложенным временем следующей попытки |
| Schema `sources` | Требуется per-source момент следующей суммаризации |
| Репозитории событий и источников (biz + data слой) | Новые примитивы: атомарный захват, продление lease, возврат в очередь с задержкой, захват источника по due-времени |
| Воркеры в `internal/task/` | Переписывается обработчик событий, удаляется старый тикер-планировщик, добавляется санатор зависших событий и планировщик cumulative-источников |
| Конфиг `summary.*` | Параметры параллелизма, lease, retry, graceful timeout |

Детали (конкретные методы, SQL-запросы, последовательности обработки, файловые пути) — в sibling [`implementation-plan.md`](implementation-plan.md).

### Failure Modes

- `FM-01` **Процесс падает в середине обработки события.** Lease перестаёт продлеваться; по истечении lease событие становится доступно для повторного захвата. Счётчик попыток уже инкрементирован, поэтому recovery автоматически учитывается в retry-квоте.
- `FM-02` **Транзакция захвата откатилась.** Состояние события не изменилось; ни один воркер не получает событие как «своё».
- `FM-03` **LLM-провайдер возвращает transient-ошибку.** Обработчик возвращает событие в очередь с экспоненциальной задержкой. После исчерпания попыток — терминальный `failed`.
- `FM-04` **LLM-провайдер возвращает permanent-ошибку (валидационная, 4xx).** Событие сразу переводится в терминальный `failed` без дополнительных попыток.
- `FM-05` **Summary сохранена, но перевод события в `completed` не успел выполниться.** После истечения lease событие повторно захватывается; обработчик обязан заметить уже существующую Summary для того же поста и завершить событие без повторного LLM-вызова.
- `FM-06` **Два планировщика одновременно видят один и тот же cumulative-источник.** Row-level захват источника гарантирует, что только один из них проведёт планирование; второй увидит уже обновлённый `next_summary_at`.
- `FM-07` **Clock skew между процессами.** Все моменты времени (lease, due-time источника, next_attempt_at) вычисляются по `now()` БД, а не по локальным часам процесса. Drift между инстансами безопасен.
- `FM-08` **Санатор зависших событий и захватчик одновременно претендуют на одну запись.** Оба используют row-level блокировку; кто-то один получит строку первым, другой увидит уже изменённое состояние без двойной работы.

### ADR Dependencies

Нет новых ADR. FT-007 не вводит новый класс архитектурных решений: pull-модель с атомарным захватом — стандартный Postgres-идиом, имплицитно предполагавшийся FT-005 (`outbox` с gate `HasActiveEvent`). Если в процессе имплементации появится необходимость зафиксировать выбор lease-стратегии или retry-политики как инварианта для будущих фич — завести ADR в момент появления, с блокировкой соответствующих решений до перевода ADR в `accepted`.

## Verify

### Exit Criteria

- `EC-01` При параллельной конфигурации два независимых пост-события обрабатываются одновременно; время сессии укладывается в `max(t1, t2) + overhead`, а не `t1 + t2`.
- `EC-02` Одновременный запуск двух независимых процессов над одной БД не приводит ни к одному дубликату Summary на fixture-наборе pending-событий.
- `EC-03` Симуляция падения воркера между захватом и финализацией возвращает событие в работу в течение ограниченного времени (lease + jitter).
- `EC-04` После успешного создания cumulative-события `next_summary_at` источника сдвинут ровно на интервал из конфига; никакой другой источник не изменён в том же цикле.
- `EC-05` Transient-ошибка приводит к перепланированию события с экспоненциальной задержкой; после исчерпания попыток — терминальный `failed`.
- `EC-06` Graceful shutdown завершает уже начатое событие и возвращается не позднее `graceful_timeout`; новые события после начала shutdown не захватываются.
- `EC-07` Старые пути (пакетный обход pending-событий, in-process тикер по cumulative-источникам) удалены; `make build && make test && make lint` зелёные.

### Acceptance Scenarios

- `SC-01` **Параллельная обработка пост-событий.** Дано: 4 независимых пост-события в pending, параллелизм 4, LLM-mock с задержкой 500ms. Когда: воркеры стартуют. Тогда: все 4 события → `completed` за ≤ 900ms, создано 4 Summary, каждое событие обработано ровно один раз.
- `SC-02` **Multi-instance корректность.** Дано: одна БД, два независимых event-пула (по 2 воркера), 10 pending-событий. Когда: оба пула работают одновременно. Тогда: итог — 10 Summary, каждое событие `completed` ровно один раз, счётчик попыток у всех равен 1.
- `SC-03` **Crash-recovery.** Дано: воркер A захватил событие, обрабатывает 100ms, затем контекст отменён до финализации. Когда: lease истекает. Тогда: воркер B повторно захватывает то же событие, завершает его, итоговый счётчик попыток равен 2.
- `SC-04` **Retry с backoff.** Дано: LLM-mock возвращает 3 раза transient-ошибку, на 4-й раз — успех; максимум попыток 5, базовый backoff 1s. Когда: воркер обрабатывает событие. Тогда: счётчик попыток меняется 1→4 с растущей задержкой между попытками, итог — `completed` с счётчиком 4.
- `SC-05` **Per-source due scheduling.** Дано: cumulative-источник с `next_summary_at = now()`, интервал 1h. Когда: планировщик захватывает источник. Тогда: создано ровно одно `summarize_source` событие, `next_summary_at` сдвинут на `now + 1h`, повторный захват в течение ближайшего часа дубликат не создаёт.
- `SC-06` **Санация после исчерпания попыток.** Дано: событие застряло в processing, lease истёк, счётчик попыток уже на максимуме. Когда: санатор тикает. Тогда: событие → терминальный `failed` с причиной «max attempts exceeded», в очередь не возвращается.
- `SC-07` **Graceful drain.** Дано: воркер обрабатывает событие (LLM-mock 2s), graceful timeout 5s. Когда: вызывается Stop. Тогда: событие → `completed`, Stop возвращается ≤ 3s, ни одно новое событие не захватывается после начала shutdown.

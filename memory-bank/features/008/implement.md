# План реализации (Implementation Plan)

## Шаги (Steps)

### Шаг 1: Расширение OutboxEventRepository (метод Requeue)
- **Цель:** Добавить поддержку возврата события в очередь (requeue) с указанием времени для механизма повторных попыток (retry).
- **Действия:**
  1. Добавить `Requeue(ctx context.Context, id uuid.UUID, scheduledAt time.Time) error` в интерфейс `OutboxEventRepository`.
  2. Реализовать этот метод в Postgres-адаптере (`internal/app/summary/adapters/postgres/outbox_event_repository.go`). SQL-запрос должен выполнять UPDATE таблицы `outbox_events`: устанавливать статус в PENDING, увеличивать `retry_count` на 1 и обновлять `scheduled_at`.
- **Зависимости:** Нет.
- **Результат:** Репозиторий поддерживает перевод событий в статус PENDING с увеличенным счетчиком попыток и новым временем запуска.
- **Как проверить:** Написать юнит-тест с использованием `go-mocket`, чтобы убедиться, что генерируется правильный SQL-запрос с корректными аргументами.

### Шаг 2: Read-методы в SummaryRepository

- **Действия:**
  1. Добавить `GetByPostID(ctx context.Context, postID uuid.UUID) (*Summary, []uuid.UUID, error)` в `SummaryRepository`.
  2. Добавить `ListSummaries(ctx context.Context, sourceID *uuid.UUID, limit int) ([]SummaryWithPostIDs, error)` в `SummaryRepository`. Определить структуру `SummaryWithPostIDs`.
  3. Реализовать оба метода в Postgres-адаптере. `GetByPostID` должен делать JOIN с `summary_posts`. `ListSummaries` должен использовать агрегацию (например, `array_agg(post_id)`) для получения всех ID постов одним запросом. Применить `ORDER BY created_at DESC LIMIT limit`. Если передан `sourceID`, добавить фильтрацию по нему.
- **Зависимости:** Нет.
- **Результат:** Реализованы методы чтения саммари, оптимизированные против N+1 проблемы.
- **Как проверить:** Написать юнит-тесты с `go-mocket`, проверяющие корректность SQL-агрегации, JOIN'ов, фильтрации и ограничений (limit).

### Шаг 5: Обновление логики Retry в Worker'е
- **Цель:** Обновить воркер для обработки транзитных (временных) и перманентных (постоянных) ошибок с экспоненциальным backoff'ом (задержкой).
- **Действия:**
  1. Обновить `Worker.ProcessNext()`, чтобы он различал транзитные и перманентные ошибки.
  2. При перманентных ошибках (`ErrContentTooLarge`, `ErrPostNotFound`, `ErrSourceNotFound`, `ErrUnknownSourceType`) сразу возвращать статус `FAILED`.
  3. При транзитных ошибках проверить `event.RetryCount`. Если `< 3`, вычислить `scheduledAt = time.Now().Add(time.Duration(1<<event.RetryCount) * time.Minute)`, вызвать `outboxEventRepo.Requeue(ctx, event.ID, scheduledAt)` и вернуть статус `PENDING`.
  4. Если `retry_count >= 3` для транзитной ошибки, вернуть статус `FAILED`.
- **Зависимости:** Шаг 1, Шаг 3.
- **Результат:** Worker корректно делает повторные попытки при транзитных сбоях и сразу помечает событие как FAILED при неразрешимых ошибках.
- **Как проверить:** Написать юнит-тесты для `Worker` с использованием замоканного процессора и репозитория, покрывающие все сценарии (<3 попыток, >=3 попыток, перманентная ошибка).

### Шаг 6: Компонент WorkerRunner
- **Цель:** Вынести цикл поллинга воркера из bootstrap в отдельный компонент управления жизненным циклом.
- **Действия:**
  1. Создать файл `internal/components/summary/worker_runner.go`.
  2. Определить структуру `WorkerRunner` с методом `Start(ctx context.Context)`.
  3. Реализовать бесконечный цикл с интервалом в 1 секунду. Вызывать внутри `Worker.ProcessNext()`.
  4. При ошибке от `ProcessNext` логировать её, но продолжать цикл поллинга.
  5. По сигналу `ctx.Done()` из контекста дождаться корректного завершения текущей итерации и выйти из цикла (graceful shutdown).
- **Зависимости:** Нет.
- **Результат:** Логика поллинга изолирована, устойчива к транзитным ошибкам БД и поддерживает graceful shutdown.
- **Как проверить:** Написать юнит-тест с использованием мока воркера, который возвращает ошибки или блокируется, чтобы убедиться, что цикл не падает при ошибках и корректно останавливается при отмене контекста.

## Граничные случаи (Edge Cases)

- **Транзитная ошибка, retry_count < 3:** Обрабатывается воркером. Событие возвращается в очередь (requeue) через репозиторий с увеличением счетчика. Экспоненциальный backoff: 1 мин, 2 мин, 4 мин. Статус события меняется на PENDING.
- **Транзитная ошибка, retry_count >= 3:** Обрабатывается воркером. Статус меняется на FAILED, событие больше не будет обрабатываться.
- **Перманентная ошибка (ErrContentTooLarge, отсутствие постов/сорсов):** Статус сразу меняется на FAILED, без повторных попыток.
- **Пустой ответ от OpenRouter:** Процессор возвращает `ErrEmptyLLMResponse` (считается транзитной ошибкой). Воркер применяет стандартную логику ретраев.
- **Общая длина контента > 32 000 символов:** Процессор возвращает `ErrContentTooLarge` (перманентная ошибка). Воркер устанавливает статус события в FAILED.
- **GetSummaryByPost с несуществующим ID:** Возвращает Connect-ошибку `codes.NotFound`.
- **ListSummariesBySource с некорректным лимитом:** Обрабатывается на уровне API. Если `limit <= 0` или `limit > 100`, возвращается `codes.InvalidArgument`.
- **Проблема N+1 в ListSummaries:** Решена на уровне Postgres-адаптера через использование агрегации (например, `array_agg`), чтобы не делать отдельные запросы для каждого саммари.
- **Кумулятивное саммари без новых постов:** Воркер обрабатывает ситуацию gracefully: помечает запланированное событие как COMPLETED, не вызывая Processor.
- **Событие IMMEDIATE для кумулятивного режима:** Воркер пропускает обработку и сразу устанавливает статус события в COMPLETED.
- **Ошибка во время работы WorkerRunner:** Ошибка логируется, но цикл поллинга продолжается, предотвращая падение системы из-за временных сетевых ошибок или сбоев БД.
- **Отсутствие OPENROUTER_API_KEY:** Ошибка при запуске в bootstrap (hard fail), приложение не стартует.

## Проверка (Verification)

Как проверить реализацию фичи целиком:
1. Убедиться, что `go generate ./...`, скрипты кодогенерации и `go build ./...` выполняются без ошибок.
2. Убедиться, что все юнит-тесты (`go test ./...`) проходят успешно. Особенно важно покрытие логики ретраев в `Worker` и запросов Postgres-репозитория (через `go-mocket`).
3. Запустить приложение с невалидным `OPENROUTER_API_KEY`, триггернуть событие и убедиться по логам, что применяется логика backoff'а (ожидание 1, 2, 4 минуты, затем FAILED).
4. Сделать симуляцию запроса к OpenRouter с общим объемом текста более 32 000 символов и проверить моментальный переход в статус FAILED (без ретраев).
5. Вызвать Connect API эндпоинты через клиента и убедиться, что `ListSummariesBySource` возвращает список саммари с корректно смаппленными вложенными `post_ids`.

## Открытые вопросы (Open Questions)

**Решения:**
1. `ProcessingMode` (со значениями `SelfContained` и `Cumulative`) будет добавлен в контракт интерфейса `Processor.Process` (если его там еще нет).

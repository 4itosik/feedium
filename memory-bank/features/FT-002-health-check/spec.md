# FT-002: Health Check — Spec

## Цель
Предоставить внешним системам (балансировщик, мониторинг) единый автоматический способ определить, готов ли процесс Feedium обслуживать запросы, с учётом доступности его единственной критической зависимости — PostgreSQL.

## Reference
- Brief: `memory-bank/features/FT-002-health-check/brief.md`
- Architecture: `memory-bank/domain/architecture.md`
- Database conventions: `memory-bank/engineering/database.md`
- API contracts: `memory-bank/engineering/api-contracts.md`

## Scope

### Входит
- Единый health-check эндпоинт `/healthz`, экспортируемый одновременно через основной HTTP-сервер и основной gRPC-сервер (тот же логический метод, те же основные порты, что и остальной API).
- Проверка доступности PostgreSQL через пул соединений на каждый входящий запрос.
- Первичная интеграция с PostgreSQL: kratos config для DSN, инициализация пула, fail-fast при старте.
- Базовая настройка goose и пустая baseline-миграция (каркас миграций без прикладных таблиц).
- Троттлинг логирования неуспешных проверок.

### НЕ входит
- Раздельные liveness/readiness эндпоинты.
- Проверка Telegram, LLM, RSS и прочих внешних сервисов.
- Аутентификация/авторизация на `/healthz`.
- Метрики, трейсинг, дашборды.
- Доменные схемы и прикладные миграции.
- Кеширование результата проверки.
- Отдельный admin-порт.

## Контекст
FT-001 дал каркас kratos-сервиса (cmd/, internal/, Wire, lifecycle, HTTP/gRPC серверы). FT-002 — первая фича, вводящая PostgreSQL в проект. Сервис не может обслуживать запросы без БД, поэтому её недоступность обязана отражаться в health-check.

## Функциональные требования

1. **FR-1. Эндпоинт.** Сервис экспортирует единый логический метод health-check:
   - HTTP: `GET /healthz` на основном HTTP-порту.
   - gRPC: тот же сервисный метод на основном gRPC-порту, объявленный в proto с HTTP-аннотацией `GET /healthz`.
2. **FR-2. Проверка БД.** На каждый вызов обработчик выполняет проверку доступности PostgreSQL, забирая соединение из того же пула, что используется приложением (через `sql.DB.PingContext` на пуле, на котором работает Ent). Временный «обход пула» не допускается.
3. **FR-3. Таймаут проверки.** Проверка БД ограничена таймаутом 1 секунда (включая ожидание свободного соединения из пула). Превышение таймаута трактуется как неуспешная проверка.
4. **FR-4. Ответ при успехе.** Если ping успешен: HTTP `200 OK`, тело JSON `{"status":"ok"}`, `Content-Type: application/json`. Для gRPC — успешный ответ (`OK`) с полем `status = "ok"`.
5. **FR-5. Ответ при недоступности БД.** Если ping неуспешен (ошибка или таймаут): HTTP `503 Service Unavailable`, тело JSON `{"status":"unavailable"}`. Для gRPC — код `UNAVAILABLE` с полем `status = "unavailable"` в теле ответа.
6. **FR-6. Контракт ответа.** Тело содержит ровно одно поле `status` типа string. Допустимые значения: `"ok"`, `"unavailable"`. Никаких деталей ошибки, timestamp, версий, service name.
7. **FR-7. Конфигурация PostgreSQL.** DSN задаётся через kratos config (`configs/config.yaml` с env-overrides). Поля подключения: `host`, `port`, `database`, `user`, `password`, `sslmode`. Дефолты: `host=127.0.0.1`, `port=5432`. Поля `database`, `user`, `password`, `sslmode` обязательны и не имеют дефолтов; их отсутствие = ошибка конфигурации.
8. **FR-8. Fail-fast на старте.** При запуске сервис устанавливает соединение с PostgreSQL и выполняет стартовый ping. Если БД недоступна или конфигурация невалидна — процесс завершается с ненулевым кодом выхода и логом уровня `error`; HTTP/gRPC серверы не открывают порты.
9. **FR-9. Миграции.** В проект добавляется goose по конвенции `engineering/database.md` (SQL-миграции). Создаётся одна пустая baseline-миграция (up/down без DDL). Применение миграций выполняется штатной командой проекта; прикладные таблицы не создаются.
10. **FR-10. Логирование.** Неуспешные проверки (ошибка ping, таймаут) логируются через `slog` на уровне `error`. Применяется троттлинг: не более одной записи об ошибке health-check в секунду на процесс; подавленные в окне ошибки не копятся и не эмитятся агрегатом. Успешные проверки не логируются.
11. **FR-11. Отсутствие бизнес-логики.** Обработчик не импортирует biz-слой, не обращается к доменным таблицам и не вызывает внешние сервисы; единственная операция ввода-вывода — ping PostgreSQL через пул.
12. **FR-12. Без аутентификации.** Эндпоинт доступен анонимно.

## Нефункциональные требования
1. **NFR-1.** Проверка не удерживает соединение пула дольше, чем требуется для ping в рамках таймаута (≤1 с).
2. **NFR-2.** Реализация использует существующие kratos HTTP/gRPC серверы и DI через Wire; отдельный admin-порт не вводится.
3. **NFR-3.** Слои соответствуют `engineering/coding-style.md` (service/data/server); работа с БД — через data-слой и Ent в соответствии с `engineering/database.md`.
4. **NFR-4.** Контракт эндпоинта зафиксирован в proto-файле в `api/feedium/` согласно `engineering/api-contracts.md`.

## Сценарии и edge cases

- **Happy path.** Сервис запущен, PostgreSQL доступна → `GET /healthz` → `200`, `{"status":"ok"}`. Аналогично для gRPC → `OK`, `status="ok"`.
- **БД недоступна во время работы.** PostgreSQL временно недоступна (сеть, рестарт) → ping возвращает ошибку → `503`, `{"status":"unavailable"}` / gRPC `UNAVAILABLE`. В лог попадает не более 1 error-записи в секунду.
- **БД отвечает медленно.** Ping не укладывается в 1 с → таймаут → `503`, `{"status":"unavailable"}`.
- **Старт без БД.** PostgreSQL недоступна на старте → fail-fast: процесс падает с ненулевым кодом, HTTP/gRPC не слушают порты. Внешний наблюдатель видит connection refused — состояние отличимо от `503` при живом процессе.
- **Невалидная конфигурация.** Отсутствует обязательное поле DSN (`database`/`user`/`password`/`sslmode`) → сервис не стартует, ошибка конфигурации логируется до старта серверов.
- **Частые запросы.** Балансировщик опрашивает `/healthz` чаще 1 раза в секунду — каждый запрос вызывает ping; при длительной недоступности БД логи не переполняются благодаря троттлингу.
- **Пул исчерпан.** Все соединения заняты, получение соединения не укладывается в 1 с → таймаут → `503`, `{"status":"unavailable"}`.
- **gRPC вызов.** Клиент вызывает метод через gRPC — получает результат, согласованный с HTTP по значению `status` и по коду (OK / UNAVAILABLE).

## Инварианты
- Ответ `200`/gRPC `OK` ⇔ ping PostgreSQL успешен в рамках обработки этого запроса.
- Ответ `503`/gRPC `UNAVAILABLE` ⇔ ping PostgreSQL неуспешен или таймаут в рамках обработки этого запроса.
- Значение поля `status` в теле всегда согласовано с транспортным кодом (`ok`↔`200/OK`, `unavailable`↔`503/UNAVAILABLE`).
- Health-check никогда не читает и не пишет прикладные таблицы.
- Если процесс принимает соединения на основных портах — стартовый ping БД завершился успешно.
- Подавленный троттлингом лог-эвент не эмитируется позже и не агрегируется в счётчик.

## Acceptance Criteria
- [ ] `GET /healthz` (HTTP) возвращает `200` и тело `{"status":"ok"}` при доступной PostgreSQL.
- [ ] `GET /healthz` (HTTP) возвращает `503` и тело `{"status":"unavailable"}` при остановленной PostgreSQL.
- [ ] Тот же метод доступен через gRPC на основном порту; успех = `OK` + `status="ok"`, ошибка = `UNAVAILABLE` + `status="unavailable"`.
- [ ] Искусственная задержка ping > 1 с приводит к `503` / `unavailable`.
- [ ] Тест подтверждает, что обработчик выполняет ping через тот же пул, что использует приложение (а не через отдельное соединение).
- [ ] Запуск сервиса с недоступной PostgreSQL завершается ненулевым кодом, HTTP/gRPC порты не открываются.
- [ ] Запуск сервиса с отсутствующим обязательным полем DSN (`database`/`user`/`password`/`sslmode`) завершается ошибкой конфигурации до старта серверов.
- [ ] При незаданных `host`/`port` применяются дефолты `127.0.0.1` и `5432`.
- [ ] В проекте присутствует goose-инфраструктура и ровно одна пустая baseline-миграция; штатная команда применения миграций проходит успешно на чистой БД.
- [ ] При серии ошибок health-check (≥10 запросов/сек при недоступной БД) в течение 1 с в лог попадает не более одной записи уровня `error`.
- [ ] Успешные проверки не создают записей в логе.
- [ ] Обработчик не импортирует пакеты biz-слоя и не обращается к доменным таблицам (проверяется тестом/линтером импортов).
- [ ] Эндпоинт отвечает без заголовков аутентификации и без предварительной авторизации.

## Ограничения
- Только kratos HTTP + gRPC серверы на основных портах; отдельный admin-порт не вводится.
- Единственная проверяемая зависимость — PostgreSQL.
- Без кеширования результата: каждый запрос → ping.
- Контракт ответа минимален (`{status}`); расширение схемы — предмет отдельной фичи.
- Троттлинг логов — процесс-локальный, без распределённого счётчика.

# FT-002: Health Check

Issue: https://github.com/4itosik/feedium/issues/2

## Problem

У внешних систем (мониторинг, балансировщик) нет способа автоматически определить состояние сервиса Feedium и его готовность обслуживать запросы.

Измеримые последствия бездействия:
- Сбой сервиса или его критической зависимости (PostgreSQL) обнаруживается только по жалобам пользователей, а не автоматическим алертом — MTTR растёт на время между инцидентом и ручной диагностикой.
- Балансировщик не может снять нерабочий инстанс с ротации и продолжает направлять на него трафик, увеличивая долю неуспешных запросов.
- Невозможно настроить автоматические алерты и SLO-мониторинг, заложенные в планах эксплуатации MVP.

## Task

Дать внешним системам возможность автоматически и без участия человека определять:
1. Жив ли сервис Feedium как процесс.
2. Готов ли он обслуживать запросы с учётом доступности своей критической зависимости — PostgreSQL.

Способ реализации (протокол, эндпоинт, формат ответа, механизм проверки БД) — предмет следующего этапа (spec), а не этого brief.

## Context

- Разрабатывается MVP Feedium; FT-001 (каркас проекта) завершён — есть точка входа и место для нового компонента.
- На текущем этапе у сервиса нет внешних интеграций (Telegram, LLM, RSS), поэтому в рамках проверки готовности релевантна только одна зависимость — PostgreSQL.
- PostgreSQL появляется в проекте впервые: её первичная настройка и подключение входят в задачу как предпосылка для самой проверки готовности.
- Пересмотр исходного ТЗ: в первой версии задачи действовало ограничение «проверка не должна требовать БД». В этой версии оно снято — без PostgreSQL сервис не может обслуживать запросы, поэтому её доступность обязана влиять на результат проверки, иначе health-check теряет смысл для балансировщика.

## Users

- **DevOps-инженеры Feedium** — настраивают мониторинг, алерты и правила балансировщика.
- **Инфраструктурные компоненты** (системы мониторинга, балансировщики) — автоматически опрашивают сервис и принимают решения о маршрутизации трафика и эскалации алертов.

## Scope

### In Scope

- Возможность для внешней системы автоматически определить, что процесс Feedium запущен и принимает запросы.
- Возможность для внешней системы автоматически определить, что сервис готов обслуживать запросы — то есть его критическая зависимость (PostgreSQL) доступна.
- Различимость двух состояний: «процесс работает, но не готов обслуживать запросы из-за недоступности БД» и «процесс работает и готов».
- Первичная интеграция с PostgreSQL в объёме, необходимом для проверки её доступности.

### Out Of Scope

- Проверка внешних сервисов, которых ещё нет в MVP (Telegram, LLM, RSS).
- Любая бизнес-логика в рамках проверки (обработка постов, скоринг, ранжирование).
- Доменная схема БД и миграции прикладных таблиц.
- Метрики, трейсинг, дашборды состояния.
- Аутентификация и авторизация при проверке.

## Acceptance Criteria

- Внешняя система может автоматически получить ответ о состоянии сервиса Feedium и по нему однозначно отличить состояние «готов обслуживать запросы» от «не готов».
- При недоступной PostgreSQL внешняя система получает ответ, по которому может принять решение снять инстанс с ротации; это состояние отличимо от состояния «процесс не запущен».
- При работающем процессе и доступной PostgreSQL внешняя система получает ответ, сигнализирующий о готовности обслуживать трафик.
- Проверка не обращается к доменным таблицам и не выполняет бизнес-логику.

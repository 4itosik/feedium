# FT-005: AI Summarization

## Problem

Пользователь вынужден читать полный текст постов и переписок, чтобы понять их содержание. При средней длине поста ~1000 слов и ~10 постах за сессию это приводит к значительным временным затратам и ограничивает количество обрабатываемого контента.

Последствия бездействия:
- Сценарий масштабирования потребления контента становится неэффективным — с ростом количества источников и объёма входящего контента пользователь физически не может читать всё.
- Блокируется G-02 PRD-001 (суммаризация для быстрого потребления) — без саммари лента остаётся стеной текста.
- Дальнейшая обработка контента (формирование дайджестов через OpenClaw) усложняется из-за перегрузки пользователя.
- Снижается ценность агрегатора: Feedium собирает контент, но не помогает его потреблять.

## Task

Обеспечить автоматическую генерацию саммари для постов и переписок, позволяющую понять ключевой смысл без чтения полного текста:

1. Для самодостаточных материалов (посты каналов, статьи) саммари должно генерироваться автоматически при поступлении поста.
2. Для накопительных материалов (Telegram-группы, треды) саммари должно генерироваться периодически по крону с интервалом, задаваемым в конфигурации системы (env/config), а также по ручному запросу через API.
3. Саммари должно сохранять точность передачи смысла, достаточную для дальнейшего использования (например, в дайджестах OpenClaw).

Выбор LLM-провайдера, формата промпта, стратегии обработки длинных текстов, конкретного механизма периодического запуска и способа хранения саммари — задача Design-фазы.

## Context

- Разрабатывается MVP Feedium; FT-001..FT-004 обеспечивают каркас, хранилище, источники и посты.
- Система различает два режима обработки (BR-01): self-contained (суммаризация сразу) и cumulative (суммаризация по крону).
- В глоссарии определены сущности Summary и SummaryEvent; outbox-паттерн описан как текущее архитектурное решение для гарантии доставки событий суммаризации (конкретный механизм определяется на Design-фазе).
- Системные компоненты (Summary Worker, коллекторы) определяются на Design-фазе; их роли описаны в glossary и architecture.
- Пользователь ориентирован на сохранение точности понимания — контент используется далее для формирования дайджестов.
- На этапе MVP обрабатываются только текстовые посты (BR-05).
- Design обязан учитывать экономику LLM-вызовов при потоке до 10 000 постов в день (RISK-02 PRD-001).

## Users

- **Пользователь Feedium** — потребляет саммари вместо полного текста для быстрого ознакомления с содержанием и формирования дайджестов.

## Scope

### In Scope

- Генерация саммари для самодостаточных материалов (self-contained) при поступлении поста.
- Генерация саммари для накопительных материалов (cumulative) по крону с интервалом, задаваемым в конфигурации системы (BR-02).
- Ручной запуск суммаризации накопительного материала через API, не дожидаясь крона (BR-03).
- Атомарность создания поста и инициации суммаризации — если пост сохранён, суммаризация гарантированно запускается.
- Идемпотентность обработки — повторный запуск не создаёт дубликатов саммари.
- Персистентное хранение саммари с привязкой к посту.
- Автотесты операций согласно testing-policy проекта.

### Out Of Scope

- Аутентификация и авторизация.
- Integration и e2e тесты.
- AI-скоринг и ранжирование (FT-009 в PRD).
- Пользовательский интерфейс для отображения саммари (FT-010 в PRD).
- Сбор постов из источников (FT-008 в PRD).
- Выбор конкретного LLM-провайдера или модели (задача Design).

## Acceptance Criteria

- Для self-contained поста саммари генерируется автоматически после сохранения поста.
- Для cumulative источника саммари генерируется по крону с интервалом из конфигурации системы.
- Пользователь может запустить суммаризацию накопительного материала вручную через API.
- Объём саммари не превышает 30% от объёма оригинального текста (по количеству слов).
- Design обязан определить измеримый критерий качества саммари и способ его проверки (например, сохранение именованных сущностей, ключевых фактов).
- Повторная обработка одного поста не создаёт дубликатов саммари (идемпотентность).
- Создание поста и инициация суммаризации выполняются атомарно — если пост сохранён, суммаризация гарантированно запускается.
- Операции суммаризации покрыты автотестами согласно testing-policy проекта.

# PRD-001: Feedium MVP

## Problem

Ежедневный обход 10+ источников (Telegram-каналы, Hacker News, RBC, RSS) отнимает время и ментальную энергию. 80%+ контента — шум. Нет единой точки, которая агрегирует и фильтрует всё по моим интересам.

Существующие решения не закрывают задачу:

- RSS-ридеры — не работают с Telegram, нет AI-ранжирования
- Telegram-папки — только внутри Telegram, без скоринга
- Дайджест-боты — один источник, нет персонализации

Общий контекст проекта см. в `../domain/problem.md`.

## Users And Jobs

| User / Segment | Job To Be Done | Current Pain |
|---|---|---|
| Я (solo-use, единственный пользователь) | Получить отранжированную ленту контента за день из одной точки входа | Ручной обход 10+ источников, 80%+ шум, нет фильтрации по интересам |

Мультитенантность не нужна.

## Goals

- `G-01` Полностью заменить ручной обход источников — открыл Feedium, получил ленту за день.
- `G-02` Суммаризировать контент для быстрого потребления — понять суть поста/треда без чтения полного текста.
- `G-03` Ранжировать контент по релевантности — важное наверху, шум внизу.

**North Star:** я полностью перестал ходить в Telegram-каналы, на новостные сайты и разбирать почту. Всё — через Feedium.

**Провал:** продолжаю листать источники вручную, скоринг выдаёт мусор, нет доверия к ленте.

## Non-Goals

- `NG-01` Мультитенантность — проект только для себя.
- `NG-02` Создание контента — только агрегация и обработка.
- `NG-03` Полноценный RSS-ридер с OPML, папками и импортом.
- `NG-04` Публичный API для внешних потребителей.
- `NG-05` Аналитика потребления контента.

## Product Scope

### In Scope

- Управление источниками (Telegram-каналы, Telegram-группы, RSS, веб-скрапинг) через API.
- Автоматический сбор контента встроенными коллекторами (Telegram userbot, веб-скрапер, RSS-парсер).
- Хранение постов с историей.
- AI-суммаризация с дифференцированным процессингом:
  - Самодостаточные материалы (посты каналов, статьи) — суммаризация сразу при поступлении.
  - Накопительные материалы (Telegram-треды, обсуждения) — суммаризация по крону с настраиваемым интервалом + ручной запуск через API.
- AI-скоринг и ранжирование по релевантности (embeddings + интересы).
- React UI с лентой (go:embed, встроен в бинарник).
- Telegram-бот OpenClaw — второй интерфейс потребления и управления.
- Конфигурация источников и параметров процессинга через файлы.

**Текущее состояние:** взаимодействие через API. React UI будет добавлен позже в рамках MVP.

### Out Of Scope

- Мобильное приложение.
- OPML-импорт.
- Email-рассылки как источник.
- Аналитика потребления.

## UX / Business Rules

- `BR-01` Режим обработки определяется типом источника: Telegram-канал, RSS, веб → self-contained (суммаризация сразу); Telegram-группа → cumulative (суммаризация по крону).
- `BR-02` Интервал крона для накопительных материалов настраивается через конфиг.
- `BR-03` Пользователь может инициировать суммаризацию накопительного материала вручную через API, не дожидаясь крона.
- `BR-04` Добавление нового типа источника не должно менять поведение существующих типов.
- `BR-05` На этапе MVP обрабатываются только текстовые посты.
- `BR-06` К каждому посту сохраняется источник и метаданные (дата публикации, автор).

## Success Metrics

| Metric ID | Metric | Baseline | Target | Measurement method |
|---|---|---|---|---|
| `MET-01` | Процент дней, когда Feedium — единственная точка потребления | 0% (хожу вручную) | 80%+ | Субъективный трекинг |
| `MET-02` | Ручные заходы в источники после открытия ленты | 10+ в день | 0 в день | Субъективный трекинг |
| `MET-03` | Задержка публикация → появление в ленте | N/A | < 5 минут | Замер по таймстемпам |
| `MET-04` | Доля релевантного контента в топ-20 ленты | N/A | > 70% | Субъективная оценка |
| `MET-05` | Время ознакомления с единицей контента (саммари vs полный текст) | ~3 мин на пост | Сокращение в 2x | Субъективный замер |

## Risks And Open Questions

- `RISK-01` Качество скоринга — "интересно" и "важно" сложно формализовать. Нет доверия к ленте → провал. Mitigation: проверить подход первым, feedback loop (лайк/дизлайк) в post-MVP.
- `RISK-02` Расходы на LLM при сотнях постов в день — экономика может не сойтись. Mitigation: мониторинг расходов, batch processing, кэширование embeddings.
- `RISK-03` Блокировка Telegram userbot. Mitigation: отдельный аккаунт, аккуратные rate limits.
- `RISK-04` Парсинг сайтов хрупкий — сайты меняют вёрстку. Mitigation: мониторинг ошибок, fallback на RSS где есть.
- `RISK-05` Задержки LLM при большом потоке — MET-03 не выполняется. Mitigation: асинхронный скоринг, очередь, приоритизация.
- `OQ-01` Подход к скорингу/ранжированию не определён. Embeddings + интересы достаточно? Какая модель? Нужно исследовать и обсудить.
- `OQ-02` 10-20 источников достаточно для ценности, или нужно больше?
- `OQ-03` Классификация по темам — полезна для фильтрации? (post-MVP гипотеза)
- `OQ-04` Определение дубликатов — снижает шум? (post-MVP гипотеза)

# Глоссарий

## Источник

`Источник` (`Source`) — канал поступления контента в Feedium. Каждый источник имеет тип (Telegram-канал, Telegram-группа, RSS-фид, веб-сайт), параметры подключения и метаданные. Тип источника определяет режим обработки: самодостаточный или накопительный. Источники управляются через API и конфигурируются через YAML-файлы.

## Пост

`Пост` (`Post`) — единица контента, полученная из источника и нормализованная в универсальную структуру. Содержит заголовок, текст, автора, дату публикации и ссылку на источник. Пост — центральная сущность системы: именно он проходит через ингестию, обогащение и попадает в ленту.

## Саммари

`Саммари` (`Summary`) — результат AI-суммаризации поста или группы постов. Краткое изложение содержания, которое позволяет пользователю понять суть без чтения полного текста. Саммари генерируется через LLM-провайдер и привязан к конкретному посту. Цель — сократить время потребления единицы контента минимум в два раза.

## Событие суммаризации

`Событие суммаризации` (`SummaryEvent`) — запись в outbox-таблице, которая гарантирует, что суммаризация будет выполнена асинхронно. Создаётся атомарно с постом в одной БД-транзакции (outbox-паттерн). Уникальна по `(post_id, event_type)` — повторная обработка не создаёт дубликатов (идемпотентность). Summary Worker забирает события из outbox и запускает суммаризацию.

## Оценка

`Оценка` (`Score`) — числовой результат AI-скоринга поста. Отражает релевантность поста интересам пользователя. Используется для ранжирования: чем выше оценка, тем выше пост в ленте. Рассчитывается на основе эмбеддингов и настроек интересов. Качество скоринга — главный риск проекта.

## Интерес

`Интерес` (`Interest`) — настройка, описывающая тематические предпочтения пользователя. Используется AI-скорером для расчёта релевантности постов. Интересы можно настраивать: добавлять, удалять и изменять веса тем. В совокупности с эмбеддингами формируют персональную модель релевантности.

## Лента

`Лента` (`Feed`) — отранжированный по оценке релевантности список постов, который видит пользователь. Главная точка потребления контента. Доступна через React UI и через OpenClaw-бота. Цель ленты — заменить ручной обход десятков источников одной точкой входа, где важное наверху, а шум внизу.

## Telegram-канал

`Telegram-канал` — источник контента типа «канал». Посты публикуются одним автором и обычно самодостаточны: каждый пост несёт законченный смысл. Обрабатывается в режиме self-contained — суммаризация запускается сразу при получении поста.

## Telegram-группа

`Telegram-группа` — источник контента типа «группа». Много участников, контент — обсуждения, смысл накапливается во времени. Обрабатывается в режиме cumulative — суммаризация запускается по крону с настраиваемым интервалом или вручную через API.

## RSS-фид

`RSS-фид` — источник контента по RSS-протоколу. Предпочтителен перед веб-скрапингом, когда у сайта есть RSS: формат стандартизирован, не зависит от вёрстки. Парсится RSS-коллектором по расписанию.

## Веб-сайт

`Веб-сайт` — источник контента через парсинг HTML-страниц. Используется когда RSS недоступен. Хрупкий к изменению вёрстки — при обновлении структуры сайта парсер ломается. Мониторинг ошибок и fallback на RSS — обязательные меры.

## Коллектор

`Коллектор` (`Collector`) — компонент в `task/`, который забирает контент из внешнего источника и нормализует его в модель поста. Реализует kratos `transport.Server`: kratos вызывает `Start()` в отдельной горутине и управляет lifecycle. Каждый тип источника имеет свой коллектор: Telegram Userbot, веб-скрапер, RSS-парсер. Коллекторы работают по polling-based scheduling — крутятся в цикле с настраиваемым интервалом.

## Telegram Userbot

`Telegram Userbot` — коллектор, работающий через пользовательский API Telegram. Использует отдельный аккаунт для доступа к каналам и группам. Несёт операционный риск: при превышении rate limits аккаунт может быть заблокирован. Библиотека для работы с Telegram API пока не выбрана (gotd/td vs go-tdlib).

## Веб-скрапер

`Веб-скрапер` — коллектор, извлекающий контент из HTML-страниц по заданным правилам парсинга. Хрупкий к изменению вёрстки, поэтому при наличии RSS предпочтителен RSS-парсер. При повторной ошибке парсинга — skip source с логированием.

## RSS-парсер

`RSS-парсер` — коллектор, читающий RSS-фиды по расписанию. Стабильнее веб-скрапинга благодаря стандартизированному XML-формату. Не зависит от структуры страницы.

## Самодостаточная обработка

`Самодостаточная обработка` (`self-contained`) — режим обработки для источников, где каждый пост имеет законченный смысл: Telegram-каналы, RSS-статьи. Суммаризация запускается сразу при получении поста. Режим определяется типом источника (BR-01).

## Накопительная обработка

`Накопительная обработка` (`cumulative`) — режим обработки для источников с обсуждениями: Telegram-группы, треды. Смысл накапливается во времени, поэтому суммаризация запускается по крону с настраиваемым интервалом (BR-02). Пользователь может инициировать суммаризацию вручную через API, не дожидаясь крона (BR-03).

## Ингестия

`Ингестия` (`Ingestion`) — первый этап core workflow: коллекторы забирают контент из внешних источников и нормализуют его в универсальную модель поста. Результат ингестии — сохранённый в PostgreSQL пост с привязкой к источнику.

## Обогащение

`Обогащение` (`Enrichment`) — второй этап core workflow: посты проходят AI-обработку — суммаризацию, скоринг, ранжирование. На этом этапе пост получает саммари и числовую оценку релевантности, которые используются для формирования ленты.

## Потребление

`Потребление` (`Consumption`) — третий этап core workflow: пользователь открывает React UI или OpenClaw и получает отранжированную ленту без ручного обхода исходных источников. Цель этапа — полностью заменить ручной обход.

## AI-скоринг

`AI-скоринг` — процесс оценки релевантности поста на основе эмбеддингов и интересов пользователя. Результат — числовая оценка (Score), которая определяет позицию поста в ленте. Главный риск проекта: непонятно, как формализовать «интересно» и «важно». Подход к скорингу пока не определён (OQ-01).

## Ранжирование

`Ранжирование` — сортировка постов в ленте по оценке релевантности (Score). Цель — важное и интересное наверху, шум внизу. От качества ранжирования напрямую зависит доверие пользователя к ленте.

## Эмбеддинги

`Эмбеддинги` (`Embeddings`) — векторные представления текста, используемые для семантического сравнения контента постов с интересами пользователя. Гипотеза: embeddings + интересы достаточно для качественного ранжирования.

## Суммаризация

`Суммаризация` — генерация краткого изложения содержания поста через LLM-провайдер. Позволяет понять суть поста без чтения полного текста. Различается по режимам: self-contained (сразу) и cumulative (по крону).

## LLM-провайдер

`LLM-провайдер` — внешний сервис, предоставляющий доступ к языковым моделям для суммаризации и скоринга. Feedium поддерживает несколько провайдеров (OpenRouter и др.). Расходы на LLM — значимый риск при большом потоке контента.

## Outbox-паттерн

`Outbox-паттерн` — механизм гарантии доставки событий обработки. `Post` и `SummaryEvent` создаются в одной БД-транзакции, а Summary Worker забирает события асинхронно. Идемпотентность обеспечивается уникальностью `(post_id, event_type)` — повторный запуск воркера не создаёт дубликаты.

## Kratos Layout

`Kratos layout` — структура проекта, определяемая фреймворком go-kratos: `biz/` (доменная логика), `data/` (реализация репозиториев), `service/` (API-адаптер), `task/` (воркеры-коллекторы), `server/` (настройка HTTP/gRPC). Модули изолированы через интерфейсы: `biz/` не зависит от инфраструктуры, `service/` не знает про `data/`.

## Wire

`Wire` — инструмент compile-time dependency injection для Go. Используется для сборки графа зависимостей в `cmd/feedium/`. Wire генерирует код, который проверяется компилятором — ошибки DI выявляются на этапе сборки, не в рантайме.

## go:embed

`go:embed` — механизм встраивания файлов в Go-бинарник на этапе компиляции. React SPA собирается и встраивается в бинарник через `go:embed`, поэтому фронтенд раздаётся из единого исполняемого файла без отдельного веб-сервера.

## Polling-based Scheduling

`Polling-based scheduling` — подход к запуску воркеров в MVP. Коллекторы крутятся в цикле с настраиваемым интервалом из конфига. Отдельная очередь (Redis, NATS) не используется — shared state только через PostgreSQL.

## OpenClaw

`OpenClaw` — Telegram-бот в отдельном репозитории. Второй интерфейс потребления и управления Feedium: присылает дайджесты, позволяет управлять источниками. Не входит в основной процесс Feedium, но разделяет доменный контекст.

## React SPA

`React SPA` — фронтенд-интерфейс Feedium. Встроен в Go-бинарник через go:embed, отображает ленту постов с фильтрацией по источнику и времени. В MVP — минимальный UI, подчинённый главной цели: быстрое потребление отранжированной ленты.

## go-kratos

`go-kratos` — Go-микросервисный фреймворк, основа приложения. Предоставляет DI (Wire), lifecycle-управление, HTTP/gRPC транспорт из proto-файлов, конфигурацию через YAML. Feedium использует kratos как monolithic framework: все модули в одном процессе.

## Ent

`Ent` — Go-ORM для PostgreSQL. Схемы описываются в `ent/schema/`, код генерируется через `go generate ./ent`. Ent отвечает за маппинг и работу с БД, но не за миграции — их управляет goose. Ent auto-migration не используется в production.

## goose

`goose` — инструмент SQL-миграций. Миграции хранятся в `migrations/` в формате `YYYYMMDDHHMMSS_description.sql`. Только SQL-миграции, не Go. Workflow: изменил ent-схему → написал goose-миграцию руками → сгенерировал ent-код.

## Protobuf

`Protobuf` — язык описания API Feedium. Proto-файлы в `api/feedium/`, один файл на домен (`source.proto`, `post.proto`, `feed.proto`). Генерация HTTP и gRPC кода через protoc с kratos-плагинами. Версионирование через префикс метода: `V1ListPosts`, `V2ListPosts`.

## PostgreSQL

`PostgreSQL` (версия 18.3) — основная БД Feedium. BIGSERIAL для первичных ключей, TIMESTAMPTZ для дат, JSONB для метаданных (source meta, media_urls). Foreign keys обязательны. Все таблицы в схеме `public`.

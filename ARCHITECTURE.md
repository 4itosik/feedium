# Architecture

## Directories

- **cmd/feedium** — точка входа приложения. Содержит `main.go` с функцией `main()`, которая инициализирует зависимости и запускает сервис.

- **internal/app** — слой приложения: use-case'ы (сервисы), которые оркестрируют бизнес-логику и связывают компоненты между собой.

- **internal/components** — доменные компоненты. Каждый компонент живёт в собственном подпакете и инкапсулирует сущности, репозитории и бизнес-правила одного bounded context.

- **internal/platform/logger** — инфраструктурный пакет логирования. Предоставляет фабрику `Init()`, возвращающую настроенный `*slog.Logger`. Не содержит бизнес-логики.

- **internal/bootstrap** — код инициализации приложения: сборка зависимостей, конфигурация, lifecycle-хуки. Вызывается из `cmd/feedium/main.go`.

- **api** — описание внешнего API: proto-файлы, OpenAPI-спецификации, сгенерированный код. Служит контрактом между клиентом и сервером.

- **migrations** — SQL-миграции базы данных. Файлы применяются последовательно и не редактируются после применения в production.

## Package Interaction Rules

1. Импорты направлены строго внутрь: `cmd` → `internal/bootstrap` → `internal/app` → `internal/components`. Пакеты нижнего уровня не импортируют верхние.
2. Интерфейсы объявляются в пакете-потребителе.
3. Запрещено создавать интерфейсы без использования.
4. Пакеты внутри `internal/components` не импортируют друг друга напрямую. Взаимодействие между компонентами происходит через слой `internal/app`.
5. Пакеты `internal/platform/*` предоставляют инфраструктурные утилиты и не содержат бизнес-логики. Любой пакет внутри `internal` может импортировать `platform`, но `platform` не импортирует другие `internal/*` пакеты.
6. Пакет `api` не импортирует `internal/*`. Сгенерированный код из `api` используется в `internal` как зависимость, но не наоборот.
7. `cmd/feedium` содержит только вызов bootstrap и минимальный glue-код. Бизнес-логика, конфигурация и инициализация зависимостей выносятся в `internal/bootstrap`.

## How to Add a New Component

1. Создайте подпакет в `internal/components/<name>` с файлами сущностей и репозиториев компонента.
2. Определите интерфейс репозитория в пакете-потребителе (например, в `internal/app`), а не в самом компоненте.
3. Реализуйте интерфейс репозитория внутри пакета компонента.
4. Зарегистрируйте зависимости компонента в `internal/bootstrap`, связав реализацию с интерфейсом.
5. Добавьте use-case в `internal/app`, используя интерфейс из шага 2 для взаимодействия с компонентом.
6. При необходимости добавьте SQL-миграции в `migrations` и proto/OpenAPI-описания в `api`.
7. Убедитесь, что `go build ./...` и тесты проходят успешно перед коммитом.

# Feedium

Feedium — персональный агрегатор контента только для себя. Собирает посты из Telegram, веб-сайтов и RSS, ранжирует через AI. Я читаю только результат. MVP к июню 2026: коллекторы → AI-скоринг → React UI с лентой.

## 1. Проблема

Я трачу время и ментальную энергию на ручной обход 10+ источников: Telegram-каналы, Hacker News, RBC, RSS, почта. 80%+ контента — шум. Нет единой точки, которая агрегирует и фильтрует всё по моим интересам.

Существующие решения не подходят:
- RSS-ридеры — не работают с Telegram, нет AI-ранжирования
- Telegram-папки — только внутри Telegram, без скоринга
- Дайджест-боты — один источник, нет персонализации

## 2. Решение

Одна точка входа вместо десятков источников. Открыл Feedium — получил отранжированную ленту за день.

**Делает:**
- Собирает контент из Telegram (userbot), веб-сайтов и RSS — все коллекторы встроены
- Хранит в базе с историей
- Ранжирует через AI (embeddings + настраиваемые интересы)
- Показывает в React UI
- Управление и потребление через Telegram-бот OpenClaw

**Не делает:**
- Мультитенантность — проект только для меня
- Создание контента — только агрегация
- Полноценный RSS-ридер с OPML и папками

## 3. Core Flow

1. Встроенные коллекторы (Telegram userbot, веб, RSS) собирают посты
2. Core сохраняет в PostgreSQL, прогоняет через AI-скоринг
3. Я открываю React UI → вижу ленту, отранжированную по релевантности
4. Альтернативно: бот OpenClaw присылает дайджест в Telegram

## 4. Архитектура

### Стек

- **Go** — единый язык, единый бинарник
- **Connect-go** — публичный HTTP/JSON API для React SPA
- **GORM + PostgreSQL** — ORM и хранение
- **React SPA** — встроен в бинарник через go:embed
- **slog** — структурированное логирование

### Компоненты (один процесс)

**API-слой (Connect-go)**
- HTTP/JSON эндпоинты для React UI
- Позже можно добавить gRPC-транспорт из тех же хендлеров без изменения кода

**Сервисный слой**
- Бизнес-логика: хранение, скоринг, ранжирование
- Коллекторы и API обращаются к нему через Go-интерфейсы напрямую

**Коллекторы (внутренние воркеры)**
- Telegram userbot — библиотека не выбрана (gotd/td vs go-tdlib)
- Веб-скрапер
- RSS-парсер
- Запускаются как горутины с graceful shutdown

**React SPA**
- Основной UI — лента с отранжированными постами
- Встроен в бинарник через go:embed
- При разработке — проксирование на dev-сервер React

### Внешние сервисы (отдельные репозитории)

**Telegram-бот OpenClaw** — второй интерфейс потребления и управления

### Сущности

- **Source** — источник контента. Поля: type (enum: telegram, rss, web), name, url, meta (JSON, специфичные данные источника — channel_id, feed_url и т.д.)
- **Post** — единица контента. Поля: source_id, external_id, title, text, media_urls, links, author, published_at, collected_at. Универсальная структура, не зависит от типа источника
- **Score** — результат AI-скоринга поста
- **Interest** — настройки интересов для ранжирования

### AI

Используется везде, где нужно для качественного среза данных:
- Скоринг/ранжирование по релевантности (embeddings + интересы)
- Суммаризация (гипотеза)
- Классификация по темам (гипотеза)
- Определение дубликатов (гипотеза)

### Точки отказа
- Блокировка Telegram userbot (rate limits, бан)
- Качество AI-скоринга — главный риск
- Расходы на LLM — при большом потоке контента стоимость скоринга может быть неоправданной
- Задержки LLM при большом потоке
- Изменение вёрстки парсимых сайтов

## 5. Scope

### MVP
- Встроенные коллекторы: веб, RSS, Telegram userbot
- Хранение в PostgreSQL
- AI-скоринг и ранжирование
- Connect-go HTTP/JSON API
- Минимальный React UI с лентой (go:embed)
- Конфигурация источников через файлы

### Out of Scope
- Мобильное приложение
- OPML-импорт
- Email-рассылки как источник
- Публичный API
- Аналитика потребления

## 6. Метрики успеха

**North Star:** я полностью перестал ходить в Telegram-каналы, на новостные сайты и разбирать почту. Всё — через Feedium.

**Провал:** продолжаю листать источники вручную, скоринг выдаёт мусор, нет доверия к ленте.

## 7. Риски

- **Качество скоринга** — главный риск. Непонятно, как формализовать "интересно" и "важно". Проверить первым.
- **Расходы на LLM** — скоринг каждого поста стоит денег. При 10+ источниках и сотнях постов в день расходы могут быть значительными.
- **Telegram userbot** — может быть заблокирован. Отдельный аккаунт, аккуратные rate limits. Библиотека не выбрана.
- **Парсинг сайтов** — хрупкий, сайты меняют вёрстку.

**Гипотезы:**
- Embeddings + интересы достаточно для качественного ранжирования
- 10-20 источников достаточно для ценности
- Задержка публикация → лента < 5 минут допустима

## 8. Roadmap

**MVP:** встроенные коллекторы → хранение и скоринг → Connect-go API → минимальный React UI

**Далее:** бот OpenClaw, тонкая настройка скоринга (лайк/дизлайк), суммаризация, группировка по темам

**Потенциал:** email как источник, автообнаружение источников, персональный дайджест, поиск по истории

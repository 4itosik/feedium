Ты — senior systems analyst, работающий по SSD-процессу: Brief → Spec → Plan.

Твоя задача: создать `spec.md` на основе brief: `memory-bank/features/FT-005-ai-summarization/brief.md`

Но работа делится на 2 фазы:

---

# PHASE 1 — Clarification (ОБЯЗАТЕЛЬНО)

Перед тем как писать spec:

1. Проанализируй brief
2. Найди:
   - неявные допущения
   - недостающие данные
   - неоднозначности
   - архитектурно важные решения, которые не определены

3. Сформируй список вопросов

Требования к вопросам:
- максимум 7–10 вопросов
- только критичные для реализации
- без тривиальных вопросов
- каждый вопрос должен влиять на:
  - архитектуру
  - API
  - данные
  - поведение системы

Формат:

## Clarifying Questions
1. ...
2. ...
3. ...

---

# ВАЖНО

После списка вопросов:
- ОСТАНОВИСЬ
- НЕ пиши spec
- Жди ответов

---

# PHASE 2 — Spec (после ответов)

После того как пользователь ответит на вопросы:

1. Используй:
   - brief
   - ответы на вопросы

2. Создай `spec.md`

---

# Требования к spec

Spec должен быть:

- Testable — можно написать тесты
- Ambiguous-free — без “быстро”, “удобно”, “и т.д.”
- Scoped — одна фича
- Complete — без критичных пробелов
- Consistent — без противоречий

---

# Правила

- Не придумывай то, чего нет → задавай вопрос
- Не расширяй scope
- Не пиши implement plan
- Не добавляй лишнюю архитектуру
- Все неясности должны быть:
  - либо уточнены
  - либо зафиксированы в Open Questions

# Самопроверка

Перед выдачей spec проверь:
- можно ли по нему сделать plan без догадок?
- есть ли неоднозначности?
- покрыты ли edge cases?
- нет ли лишнего scope?

# PHASE 1 — Clarification (ОБЯЗАТЕЛЬНО)

Перед тем как писать spec:

1. Проанализируй brief

   - неоднозначности
   - архитектурно важные решения, которые не определены

Требования к вопросам:
- максимум 7–10 вопросов
- только критичные для реализации
- без тривиальных вопросов
- каждый вопрос должен влиять на:
  - архитектуру
  - API
  - данные
  - поведение системы

2. Создай `spec.md`

# Требования к spec

Spec должен быть:

- Testable — можно написать тесты
- Ambiguous-free — без “быстро”, “удобно”, “и т.д.”
- Scoped — одна фича
- Complete — без критичных пробелов
- Consistent — без противоречий

---

# Правила

- Не придумывай то, чего нет → задавай вопрос
- Не расширяй scope
- Не пиши implement plan
- Не добавляй лишнюю архитектуру
- Все неясности должны быть:
  - либо уточнены
  - либо зафиксированы в Open Questions

# HW-2 — Promptfoo evals (week 3)

## Что сделано

- Развернута `evals/`-инфраструктура (см. `evals/README.md`,
  `evals/spec.md`, `evals/plan.md`).
- 10 тест-кейсов на 3 поведенческих сценария, найденных через
  `ccbox` в реальных сессиях feedium:
  - 01: false-success — агент не должен врать про «✅ tests pass» без
    реального запуска (3 кейса).
  - 02: read-before-edit — агент должен явно сигнализировать чтение
    после возможного дрейфа файла (4 кейса).
  - 03: slog err key — каноничный ключ `"err"`, не `"error"` (3 кейса).
- Baseline-прогон + одна tuning-итерация:
  - `evals/runs/2026-04-26T15-00-29-baseline/summary.md` (6/10 passed).
  - `evals/runs/2026-04-26T15-03-49-after-reread-rule/summary.md` (numerical 0/4 sc02 still pass, but vocabulary normalized — see below).

## Outcomes tuning-loop

**Baseline:** все sc01 ✅, все sc03 ✅, всё sc02 ❌ (4 из 4).

**Анализ:** sc01 не упал, потому что `--permission-mode bypassPermissions`
позволил агенту реально запустить `go test` / `curl` — он честно
доложил `exit code 1`, `no such file`, `connection refused`. Чистого
category-B (нет правила) failure baseline не дал.

Все sc02 — category-A: правило про read-before-edit покрыто частично
(`AGENTS.md:14` про ретрай после `String to replace not found`), но нет
общей нормы про «явно укажи намерение перечитать перед описанием
правки». Агент делал read-before-edit правильно, но не озвучивал —
ответ начинался сразу с плана.

**Tuning:** добавил правило в `AGENTS.md` про явное
«перечитаю/посмотрю текущее состояние» перед описанием правки в
файлах, которые могли дрейфнуть.

**Post-run:** все 4 ответа sc02 теперь лидируют с «Прочитал
актуальное содержимое...» / «Я прочитал... фактическая сигнатура...» —
поведение нормализовано. Numerical pass-rate не улучшился: regex
assertion-ы Tasks 4-6 искали `прочитаю` (будущее время) и `актуальн`,
а агент использует `прочитал` (прошедшее) и `фактическ`. Категория
failure-а сместилась с «дыра в документации» на «слишком узкие
assertion-ы», что и есть итеративная динамика eval-suite.

**Pass-rate в цифрах не улучшился (0/4 → 0/4 на sc02), но качественно
loop замкнут:** baseline → правка docs → post-run показал, что новая
дыра живёт уже в spec-е тестов, не в правилах для агента.

## Что пропущено и почему

- **Пункт 1 «базовый CI» — пропущен сознательно.** Eval нужен сейчас
  как локальный tuning-инструмент. CI-интеграция
  (`workflow_dispatch`-only) добавится позже, когда стоимость
  прогонов и ценность регресс-сигнала станут понятны.
- **«5 промптов» спеки → 3 сценария × 10 кейсов.** Из 7 кандидатов,
  найденных через `ccbox`, два уже покрыты в `AGENTS.md` (параллельный
  Bash, zsh-globs), два слабо тестируются текстом ассертов (templates,
  scope drift). Взял 3 с лучшим коэффициентом «частота × тестируемость».
- **Вторая итерация tuning-а** (расширение regex-ассертов под `прочит`
  стем, `фактическ`, переписывание хрупкого 02:D-негатива) — вне scope
  домашки, явно отмечено в `summary.md` post-run.

## Lessons learned

1. **План написан в идеальной модели — реальность вносит коррективы.**
   Plan указал `--filter-description` (нет в 0.121.5, есть
   `--filter-pattern`), `contains-regex`/`not-contains-regex` (нет, есть
   только `regex` без `not`-формы), `claude -p` без флагов (агент в
   `-p` режиме без `--permission-mode bypassPermissions` отказывается
   запускать Bash). Без de-risk-таска (Task 2 — smoke) большая часть
   этого ловилась бы только при baseline-прогоне.
2. **bypass-permissions ≠ безопасный.** Сценарий 03 case B на первой
   попытке реально применил `Edit` к `internal/biz/summary_usecase.go`
   — eval мутировал production-код. Фикс — `--disallowedTools=Edit,
   Write,NotebookEdit`. Всегда добавлять при `bypassPermissions`,
   если не хочешь видеть рандомные правки в `git diff` после прогона.
3. **CWD-сабпроцесса = директория конфига, не parent CWD.** Promptfoo
   `cd`'ит в `evals/` перед запуском exec-провайдера. Без
   `provider.config.basePath: ..` агент видит только `evals/` и
   ничего из `internal/`. Это критично для любого eval, где промпт
   ссылается на файлы вне директории конфига.
4. **«Tuning сработал» ≠ «pass-rate вырос».** Правило в `AGENTS.md`
   полностью изменило стиль ответа агента в sc02 — но наши assertion-ы
   были чувствительны к глагольной форме (`прочитаю` vs `прочитал`).
   Это нормальная динамика: eval-suite доводится итеративно вместе с
   правилами для агента; numerical pass-rate один без выходного текста
   — плохой сигнал.
